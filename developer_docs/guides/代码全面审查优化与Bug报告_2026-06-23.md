# SunPilot 代码全面审查优化与 Bug 报告

审查日期：2026-06-23  
审查范围：当前工作区代码，只读审查；未修改业务代码。重点覆盖 Agent Runtime、WebSocket/前端聊天链路、模型路由、工具执行、上下文/记忆、Digital World、API 与测试契约。

## 1. 总体结论

当前代码已经具备比较完整的 Agent Loop 主链路：`AgentService -> AgentLoopEngine -> ContextBuilder -> IntentRouter -> ToolDecisionEngine -> ResponseComposer/AssistantMessageStream`，并且 daemon 装配层已经接入了模型路由、工具检索、审批、Trace、记忆写入、Digital World seed 和 Agent Run 完成回调。

但从“真实可用性”看，仍有几类需要优先处理的问题：

- P0：前端默认选择 `seed`，后端默认模型是 `dp`，且 `seed` 只有配置 API key 时才可用；默认聊天可能在未配置 Seed 时直接失败。
- P0：Digital World 的任务状态机在 `work_on` 启动异步 Agent 后仍会继续走到任务完成收尾，导致任务和数字生命状态提前完成。
- P0：流式 delta 的 `deltaIndex` 在 `AssistantMessageStream` 中生成，但通过 `AgentService -> JsonRpcRouter` 的实时 onDelta 路径被丢弃，前端去重逻辑无法生效。
- P1：1688/图片搜索附件校验的实现与测试契约不一致，当前实现允许“无当前图片附件”继续进入 Agent Loop，但测试仍要求硬拒绝。
- P1：前端会话排序假设 `updatedAt` 永远存在，测试中已经触发 `Cannot read properties of undefined (reading 'localeCompare')`。
- P1：测试套件存在明显漂移，部分测试仍断言旧文案或旧行为，降低了回归保护可信度。

## 3. P0 Bug

### P0-1：前端默认 Seed，但后端默认 DP，未配置 Seed 时默认发送会失败

证据：

- 后端模型接口返回 `defaultModelId: "dp"`，并且 `seed.available` 取决于 `SUNPILOT_SEED_LLM_API_KEY`：`packages/api/src/http/register-routes.ts:119`、`packages/daemon/src/server.ts:212`。
- daemon 的 `ModelRouter` 只有在 `seedApiKey` 存在时才注册 `seed` route：`packages/daemon/src/composition-root.ts:214`、`packages/daemon/src/composition-root.ts:242`。
- 前端 `ChatComposer` 内部默认 `const [model, setModel] = useState("seed")`，发送时 `useChat.send()` 也默认 `modelId: modelId ?? "seed"`：`packages/web/src/pages/ChatPage/components/ChatComposer/ChatComposer.tsx:124`、`packages/web/src/pages/ChatPage/hooks/useChat.ts:1433`。
- `ModelRouter.streamChat()` 在 request.modelId 找不到 route 时会直接 throw：`packages/core/src/agent-kernel/model-router.ts:181`。

影响：

在只配置 DeepSeek/DP、未配置 Seed API key 的常见环境中，用户不改下拉框直接发送会让所有 LLM purpose 被强制路由到不可用的 `seed`，最终变成模型路由错误。

建议：

- 前端启动时读取 `/v1/models`，默认使用 `defaultModelId`，禁用 unavailable model。
- `useChat.send()` 不应硬编码 `seed` fallback；未传入时应让后端走默认路由。
- 后端可在 `chat.send` 层对 unavailable model 返回更明确的业务错误，避免被包装成内部错误。

### P0-2：Digital World `work_on` 异步 Agent 后任务会被提前标记完成

证据：

- `TaskExecutor.executeAction()` 的 `work_on` 分支调用 `agent.startChatCommand()` 后保存 `agentRunId`，把 being 设为 `working`，然后 `return`：`packages/platform/src/digital-world/task-executor.ts:196`。
- 但 `executeTask()` 的外层循环没有识别“等待 Agent Run 完成”的暂停状态，`executeAction()` 返回后继续下一轮，最后清空 `currentTaskId/currentActionId` 并把 task 标记为 `completed`：`packages/platform/src/digital-world/task-executor.ts:75`、`packages/platform/src/digital-world/task-executor.ts:114`。
- daemon 确实订阅了 `agent.run.completed/failed/cancelled` 并调用 `onAgentRunCompleted()`：`packages/daemon/src/server.ts:226`。这意味着正确设计应该是暂停并等待回调，而不是当前执行线程继续收尾。

影响：

数字生命可能显示“任务完成/已休眠”，但 Agent Run 仍在执行或等待审批。后续 `onAgentRunCompleted()` 再次推进剩余 action，会造成状态重复推进、产物晚到、失败状态覆盖不一致等问题。

建议：

- 给 `executeAction()` 返回明确状态，例如 `completed | waiting_agent | failed`。
- `executeTask()` 遇到 `waiting_agent` 时停止后续 action，并保持 task 为 `running`。
- 只有 `onAgentRunCompleted()` 才能继续执行 `work_on` 后面的 action 或完成任务。

### P0-3：实时流式 delta 丢失 `deltaIndex`，前端去重保护失效

证据：

- `AssistantMessageStream.appendText()` 会生成并发出 `deltaIndex`：`packages/core/src/agent-kernel/assistant-message-stream.ts:141`。
- `AgentService` 的 `AgentStreamDelta` 类型没有 `deltaIndex` 字段：`packages/core/src/agent/agent.service.ts:29`。
- `AgentService` 从 raw eventBus 转发 onDelta 时只取 `messageId/partId/delta`，没有取 `deltaIndex`：`packages/core/src/agent/agent.service.ts:492`。
- `JsonRpcRouter` 包装 `agent.message.part.delta` 时也没有带 `deltaIndex`：`packages/api/src/ws/json-rpc-router.ts:94`。
- 前端 reducer 只有在 `deltaIndex` 是 number 时才执行去重：`packages/web/src/pages/ChatPage/hooks/useChat.ts:300`。

影响：

服务端已经实现了 delta 序号，但实时发送路径把它丢了。当前前端对重复/乱序 delta 的防护只在理论上存在，实际 live streaming 仍可能重复 append。由于 delta 不持久化，replay 主要靠 completed 消息修正，用户正在观看流式输出时仍会看到异常。

建议：

- `AgentStreamDelta` 增加 `deltaIndex?: number`。
- `AgentService` onDelta 和 `JsonRpcRouter` payload 都透传该字段。
- 增加一条从 `AssistantMessageStream -> AgentService.onDelta -> Web reducer` 的端到端测试。

## 4. P1 Bug 与契约漂移

### P1-1：1688 图片附件校验的实现和测试预期已经分叉

证据：

- 当前 `assertUsableImageAttachments()` 明确写了：如果当前请求没有图片附件，不拒绝，因为 `ToolArgumentBuilder` 会从历史上下文解析图片：`packages/core/src/agent/agent.service.ts:102`。
- 但 `agent.service.test.ts` 仍要求“1688 搜索无图片附件”直接 reject，错误码为 `IMAGE_ATTACHMENT_REQUIRED`：`packages/core/src/agent/agent.service.test.ts:495`。
- 当前测试结果也证明实现已经 resolved completed，而不是 rejects。

影响：

这里不一定是产品 bug，更像“产品策略变了但测试没更新”。如果目标是支持引用历史图片，则测试应该改成覆盖历史图片解析；如果目标是当前请求必须上传图片，则实现需要恢复硬拒绝。

建议：

- 明确产品规则：当前轮必须带图，还是允许历史图片。
- 若允许历史图片，应把测试改为三类：当前无图但历史有图可继续、当前无图且历史无图需澄清、当前图片引用缺失需拒绝。
- 错误 message 也要统一，当前实现是“上传的图片缺少可用引用，请重新上传后再试。”，测试期望是“图片尚未上传完成，缺少可用的图片链接。请等待上传完成后再试。”

### P1-2：会话排序对 `updatedAt` 缺少防御，测试已触发页面崩溃

证据：

- `useConversations` 排序直接调用 `b.updatedAt.localeCompare(a.updatedAt)`：`packages/web/src/pages/ChatPage/hooks/useConversations.ts:247`。
- 测试输出中 `App.test.tsx` 触发 `Cannot read properties of undefined (reading 'localeCompare')`，React Router 进入默认 ErrorBoundary。

影响：

只要 API 返回旧格式、mock fixture 缺字段、乐观创建对象缺字段，整个聊天页都会崩溃。这个问题虽由测试触发，但属于真实防御性不足。

建议：

- Conversation 类型层面让 `updatedAt` 必填并修正所有 mock/fixture。
- UI 排序层也加防御 fallback，例如 `updatedAt ?? createdAt ?? ""`。
- 添加缺字段 fixture 的回归测试，确保页面降级排序而不是崩溃。

### P1-3：ChatComposer 测试仍断言旧模型文案

证据：

- 组件当前模型选项是 `DeepSeek-4.0-flash` 和 `Seed-lite`：`packages/web/src/pages/ChatPage/components/ChatComposer/ChatComposer.tsx:90`。
- 测试仍断言 `Seed-pro`：`packages/web/src/pages/ChatPage/components/ChatComposer/ChatComposer.test.tsx:187`。

影响：

这是低风险测试漂移，但它暴露了模型选择器没有接入 `/v1/models` 的更大问题。文案硬编码会继续和后端模型目录漂移。

建议：

- 测试先改为当前真实文案或 role-based 查询。
- 更推荐把模型选项从后端模型目录加载，测试 mock `/v1/models`。

### P1-4：WebSocket 订阅模型对“非发送方实时观察”支持不足

证据：

- `ConnectionRegistry.interestedSockets()` 只向订阅了 runId、`*` 或 conversationId 的 socket 推送：`packages/api/src/ws/connection-registry.ts:52`。
- 当前 `useChat` 发送消息依赖 `chat.send` 的 streamHooks 直推 sender socket，并没有在普通会话打开后发送 `conversation.subscribe`：`packages/web/src/pages/ChatPage/hooks/useChat.ts:1001`。
- HTTP replay 会拉历史事件，但这不是 live subscribe：`packages/web/src/pages/ChatPage/hooks/useChat.ts:1325`。

影响：

当前发送者可以收到直推事件，但另一个已打开同一会话的窗口、调试面板或恢复后的连接，可能只靠轮询/replay，不能实时跟随运行中事件。

建议：

- 当前会话激活时通过 WebSocket 发送 `conversation.subscribe`，离开时 unsubscribe。
- 断线重连后用 `lastSeenSequence` replay，再恢复 subscribe。
- 保留 `chat.send` streamHooks 作为 sender 快速路径，但不要让它成为唯一实时路径。

## 5. P2 优化点

### P2-1：ContextBuilder 中关键上下文源失败目前只 console.error 后降级为空

证据：

- `listMessages/searchMemories/embedText/listArtifacts/listToolResults/listSkills` 使用 `Promise.allSettled`，失败后 `console.error` 并返回空数组或 undefined：`packages/core/src/agent-kernel/context/context-builder.ts:276`。

影响：

这样能提升可用性，但对关键源（例如 conversation messages）来说，静默降级为空可能让模型在错误上下文下回答。用户只看到“回答不对”，看不到“历史加载失败”。

建议：

- 区分关键源和非关键源：消息历史失败至少发 `agent.error` warning 或 trace span。
- 对 skill catalog 失败可以继续 no-tool，但应在 UI/RunDebugPanel 中可见。

### P2-2：模型调用持久化失败被后台 catch，只累计内存计数

证据：

- `ModelRouter.streamChat()` 中 `modelCallRecorder.create(...).catch(() => { this.persistFailures++; })`：`packages/core/src/agent-kernel/model-router.ts:245`。

影响：

这不会影响用户请求，但模型调用审计和成本统计可能悄悄丢失。`persistFailures` 目前没有明显外部暴露路径。

建议：

- 把 persist failure 写入 trace 或 diagnostics。
- 在 RunDebugPanel 展示 model-call persist failure 计数。

### P2-3：Digital World 前端空数据自动 fallback mock，生产态容易掩盖后端问题

证据：

- `useDigitalWorldBootstrap()` 在 API 返回空 nodes 或无 being 时直接使用 `mockNodes/mockEdges/mockBeing`，API 失败也 fallback mock：`packages/web/src/features/digital-world/hooks/useDigitalWorldBootstrap.ts:74`、`packages/web/src/features/digital-world/hooks/useDigitalWorldBootstrap.ts:87`。

影响：

开发体验更顺，但真实环境后端 seed 失败、数据库为空、接口异常时，用户仍看到 mock 世界，容易误判后端已经工作。

建议：

- 仅在 `import.meta.env.DEV` 或显式配置时 fallback mock。
- 生产态显示空状态/初始化失败，并提供重试或创建默认数字生命入口。

### P2-4：测试环境缺少 PixiJS/canvas mock

证据：

- Web 测试输出中 `HTMLCanvasElement.prototype.getContext` not implemented，来源于 PixiJS 导入链。

影响：

任何引入 `DigitalWorld` 或 `App` 的测试都可能被 canvas 环境噪音污染，掩盖真正断言失败。

建议：

- 在 `packages/web/src/test-setup.ts` 中 mock `HTMLCanvasElement.prototype.getContext`。
- 或对 PixiJS-heavy 组件做模块 mock，Canvas 行为交给 Playwright/e2e。

## 6. 建议修复顺序

1. 先修 `seed` 默认模型问题：这是用户默认发送路径，会直接影响聊天可用性。
2. 修 Digital World `work_on` 暂停/恢复状态机：这是任务闭环的核心正确性问题。
3. 补 `deltaIndex` 透传：修复 live streaming 去重契约。
4. 明确 1688 图片策略并同步测试：避免继续在“实现已变、测试旧了”的状态下迭代。
5. 修 `useConversations` 的 `updatedAt` 防御和测试 fixture。
6. 补 WebSocket conversation subscribe，提升多窗口/恢复场景实时性。
7. 整理测试环境：canvas mock、模型文案测试、测试命令使用方式。

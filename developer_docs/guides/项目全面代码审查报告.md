| 严重度 | 数量（约） | 说明 |

**核心结论**：

1. **SunPilot 的包边界与分层总体健康**：protocol 作为 agent 域类型唯一源；依赖图无环；prompt-injection 检测器与 trace manager 已真实接入；docker healthcheck、pgvector 镜像到位。

   - **工程化**：`evals` workspace 引用不存在的包；4 个包把测试编译进 dist；无 CI、无 precommit、无 typecheck、无 TS project references。

## 三、Critical 严重问题详解

### C1 — Replanner `dependsOn` 引用永不匹配

- **文件**：`packages/core/src/agent-kernel/planning/replanner.ts:266,449`
- **描述**：新增步骤的 `dependsOn` 写成 ``[`plan_step_newgoal_${i - 1}`]`` 与 ``[`plan_step_verify_${i - 1}`]``，但实际步骤 ID 使用 `crypto.randomUUID()` 生成，二者永不相等。
- **影响**：重规划生成的计划步骤依赖无法解析，`PlanValidator` 的依赖检查要么误判为缺失、要么被 `rebuildDependsOn` 强制覆盖，破坏执行顺序语义；多步链式工具可能乱序执行。
- **建议**：在创建步骤时缓存其真实 ID，并以该 ID 构造 `dependsOn`；或在生成后做一次 ID 重写映射。

### C2 — SkillToolExecutor 未透传 AbortSignal

- **文件**：`packages/core/src/agent-kernel/execution/skill-tool-executor.ts:90`
- **描述**：`this.deps.runSkill(step)` 调用未传入 `input.signal`。
- **影响**：用户取消 run 时，底层 skill 执行不会收到 abort 信号，长任务继续运行，浪费资源并可能写出半成品 artifact；`supportsAbort` 形同虚设。
- **建议**：将 `input.signal` 透传到 `runSkill`，并在 skill 实现中正确响应。

### C3 — ModelRouter 流式 fallback 输出错乱

- **文件**：`packages/core/src/agent-kernel/model-router.ts`（`streamChat`）
- **描述**：主模型在中途（已 yield 部分 delta）失败时，fallback 分支会从**开头**重新流式生成新内容，导致下游收到「主模型片段 + fallback 完整重述」的混合输出。
- **影响**：用户看到重复、断裂或语义矛盾的回复；AssistantMessageStream 的 `content` 拼接出错；token 计费与上下文污染。
- **建议**：fallback 时清空已产出内容并向流发送「重置/丢弃」标记，或要求 fallback 从空上下文重新开始并通知下游覆盖已写入的 text part。

### C4 — deduplicateResults 丢失 sources

- **文件**：`packages/core/src/agent-kernel/context/embedding-service.ts`（`deduplicateResults`）
- **描述**：当新项分数更高替换已存在项时，`sources` 集合被**仅用新来源重建**，原先累积的来源丢失。
- **影响**：检索溯源（provenance）信息残缺，影响审计与 MMR/多跳召回的去重判定；可能导致同一来源被重复计入。
- **建议**：替换时执行 `new Set([...existing.sources, ...newItem.sources])` 合并而非覆盖。

### C5 — memory-policy 不可达分支

- **文件**：`packages/core/src/agent-kernel/memory/memory-policy.ts:277`
- **描述**：外层 `if (existing.relevance >= 0.7)` 已限定，内层 `if (existing.relevance < 0.5 && existing.relevance >= 0.3)` 永假。
- **影响**：该分支内本应处理的「中等相关旧记忆」策略永远不执行，可能掩盖作者意图，逻辑与设计不符。
- **建议**：核对设计意图；若条件写错应修正阈值，若为残留则删除并补充正确分支。

### C6 — MemoryRetryWrapper `lastPartial` 未赋值

- **文件**：`packages/core/src/agent-kernel/memory/memory-retry.ts`
- **描述**：`lastPartial` 声明后从未被赋值，重试耗尽时返回的「部分结果」实为 `undefined`。
- **影响**：声明的部分结果回退功能完全失效；调用方拿不到降级数据，可能在记忆写入失败时直接抛错，影响对话连续性。
- **建议**：在每次成功（含部分成功）的 attempt 后保存 `lastPartial = result`，并在最终失败时返回它。

### C7 — `findRelated` 违反 PostgreSQL DISTINCT + ORDER BY 语法

- **文件**：`packages/storage/src/postgres/postgres.memory.repository.ts:340-352`
- **描述**：`findRelated` 查询使用 `SELECT DISTINCT m.*, 0.5 AS score, 0.5 AS relevance ... ORDER BY mr.established_at DESC`。PostgreSQL 规定：使用 `SELECT DISTINCT` 时，`ORDER BY` 表达式必须出现在选择列表中。`mr.established_at` 既不在 `m.*` 中，也未显式列出。
- **影响**：调用 `findRelated` 时 PostgreSQL 抛出 `ERROR: for SELECT DISTINCT, ORDER BY expressions must appear in select list`，方法完全不可用。任何依赖记忆关联图检索的功能（supersedes/contradicts/resolvedBy 等）将完全失效。
- **建议**：将 `mr.established_at` 加入选择列表，或改用 `GROUP BY m.id` + `MAX(mr.established_at)`，或去掉 `DISTINCT` 改用子查询去重。

### C8 — 数字世界表缺失外键约束

- **文件**：`packages/storage/src/migrations/020_digital_world.sql`
- **描述**：以下列未声明 FOREIGN KEY：
  - `digital_beings.current_node_id`、`target_node_id`、`home_node_id`、`current_task_id`、`current_action_id`、`current_run_id`、`conversation_id`
  - `world_actions.from_node_id`、`to_node_id`、`agent_run_id`
  - `world_artifacts.run_id`、`location_node_id`
- **影响**：
  1. 数据完整性无保障 — 可写入指向不存在记录的 ID；
  2. 删除节点/任务/会话时无法级联清理，留下大量悬空引用（与会话删除 bug 联动）；
  3. 任何按这些列 JOIN 的查询可能返回意外结果。
- **建议**：补全所有 FK 约束，并加上 `ON DELETE SET NULL` 或 `ON DELETE CASCADE` 策略。

### C9 — 本地 token 鉴权完全未实现

- **文件**：`packages/daemon/src/server.ts:81,183-187`；`packages/storage/src/paths.ts:25,60,137`
- **描述**：README 第 78 行声称"Local token auth is disabled during the current test phase"，暗示存在可被启用的鉴权机制。但实际审查发现：
  1. `SunPilotConfig.security.requireLocalToken`（paths.ts:25,60,137）是唯一与 token 鉴权相关的字段，但全工作区仅 paths.ts 自身与测试引用它，**daemon、api、ws 中无任何代码读取该字段来执行鉴权**——它是死配置。
  2. `createDaemon` 唯一的访问控制是 `onRequest` 钩子里的 `isAllowedLocalOrigin`（server.ts:183-187），而该函数在 `origin` 为空时直接 `return true`（server.ts:81）。任何非浏览器客户端（curl、本地脚本、同机其它进程）不发 Origin 头即可无限制访问全部 REST API 与 `/v1/ws`。
  3. WebSocket 升级同样仅靠 `isAllowedOrigin`（ws.ts:124），且 Origin 是浏览器自报头，非安全边界。
- **影响**：daemon 是可执行 skill、读写文件、发起网络请求、读取 secrets 的 Agent 运行时。同机任何进程都能驱动它执行高危操作。一旦通过反向代理暴露（README 明确支持 `tradeagent.asia`），且代理未额外鉴权，公网即可调用。
- **建议**：实现真正的本地 token 鉴权：daemon 启动时生成随机 token 写入 `~/.sunpilot/runtime/token`（0600），所有 HTTP/WS 请求需带 `Authorization: Bearer <token>`；`requireLocalToken` 默认改为 `true`；Origin 缺失时不应放行。

### C10 — HTTP 全路由无鉴权无限流

- **文件**：`packages/api/src/http/routes/*`、`packages/daemon/src/server.ts`
- **描述**：所有 REST 路由（`/v1/conversations`、`/v1/chat`、`/v1/digital-world`、`/v1/memory`、`/v1/runs`、`/v1/skills`、`/v1/approvals`、`/v1/artifacts`、`/v1/audit`、`/v1/settings` 等）均无任何鉴权中间件、无限流。CORS 配置 `cors({ origin: allowedOrigins, credentials: true })` 但 `allowedOrigins` 函数对 Origin 缺失返回 `true`（放行）。
- **影响**：可被 DoS（无速率限制）；可被未授权访问；跨域攻击面大。
- **建议**：补全局 token 鉴权中间件；引入 `@fastify/rate-limit`；CORS `origin` 函数对 Origin 缺失应拒绝。

### C11 — WS 优雅停机不排空 + Origin 检查薄弱

- **文件**：`packages/daemon/src/ws.ts:124`、`packages/daemon/src/server.ts`
- **描述**：
  1. `wsServer.close()` 不会主动关闭既有连接，daemon 收到 SIGTERM 后若连接未自行断开，进程会挂起或被强杀，丢失流式中途数据。
  2. WS 升级时仅调用 `isAllowedOrigin`，且 Origin 缺失放行（同 C9）。
- **影响**：优雅停机名不副实；未授权 WS 连接可建立。
- **建议**：`server.close()` 时遍历 `wsServer.clients` 并 `close(1001)`；为 WS 升级增加 token 鉴权。

### C12 — TaskExecutor 非原子领任务

- **文件**：`packages/platform/src/digital-world/task-executor.ts`
- **描述**：领任务流程为「SELECT pending → UPDATE to running」，无事务/行锁。
- **影响**：并发 worker 可同时领到同一任务，导致重复执行、artifact 双写、状态错乱。
- **建议**：改为 `UPDATE ... WHERE status='pending' RETURNING *` 单条原子语句。

### C13 — TaskService `void executeTask()` 火烧异步

- **文件**：`packages/platform/src/digital-world/task.service.ts`
- **描述**：`void executeTask(task)` 丢弃 Promise，拒绝错误无人处理。
- **影响**：任务执行失败无任何错误恢复、无审计、无状态回滚；任务永远卡在 running。
- **建议**：`await executeTask(...).catch(err => markFailed(task.id, err))` 或显式记录 metric。

### C14 — OSS 直传使用 HMAC-SHA1 弱签名

- **文件**：`packages/api/src/storage/oss-client.ts`
- **描述**：客户端签名使用 HMAC-SHA1（OSS v1 签名）。
- **影响**：SHA1 已被 NIST 弃用，存在理论碰撞风险；现代 OSS 推荐 v4 签名（HMAC-SHA256）。
- **建议**：迁移到 OSS V4 签名；或改用 STS 临时凭证 + 客户端直传。

### C15/C16 — Alert `title` prop 误用

- **文件**：
  - `packages/web/src/pages/ChatPage/components/OfflineBanner/OfflineBanner.tsx:8`
  - `packages/web/src/pages/ChatPage/components/ErrorMessageCard/ErrorMessageCard.tsx:14`
- **描述**：Ant Design 的 `Alert` 组件**没有 `title` prop**，主标题对应的 prop 是 `message`。`title` 会被 React 当作未知 DOM 属性静默丢弃。
- **影响**：离线横幅与错误卡片的标题完全不显示，用户只看到描述文字。
- **建议**：将 `title=` 改为 `message=`。

### C17 — UserMessage 不回退 `dataUrl`

- **文件**：`packages/web/src/pages/ChatPage/components/UserMessage/UserMessage.tsx:21`
- **描述**：附件渲染判断条件为 `isImageType(att) && att.url ?`，仅检查 `att.url`。但 `useFileAttachments` 在 OSS 不可用时会为小图片（≤4MB）生成 `dataUrl` 回退，此时 `att.url` 为空但 `att.dataUrl` 有值。
- **影响**：图片附件被错误地走 `else` 分支渲染为文件图标，无法预览。
- **建议**：渲染条件改为 `isImageType(att) && (att.url || att.dataUrl)`，并 `src={att.url ?? att.dataUrl}`。

### C18 — useConversations 双重请求 + 无 AbortController race

- **文件**：`packages/web/src/pages/ChatPage/hooks/useConversations.ts:183-195,263-268`
- **描述**：
  1. `selectConversation` 内先 `setActiveConversationId(id)` 再 `await getConversationMessages` 再 `setMessages`。
  2. 同时存在另一个 `useEffect` 监听 `[activeConversationId, enabled, request]`，当 `activeConversationId` 变化时**再次**发起 `getConversationMessages` 请求。
  
  结果：每次点击会话触发**两次** `GET /v1/conversations/{id}/messages`。快速切换会话时，前一个请求的响应可能晚于后一个到达，导致 `setMessages` 用旧会话的消息覆盖新会话的消息。两个请求均无 `AbortController`。
- **影响**：快速切换会话时消息列表串台/错乱；网络请求翻倍。
- **建议**：二选一删除其中一个 fetch；同时在 fetch 中加入 `AbortController`，在 `id` 变化或组件卸载时 `abort()` 旧请求。

### C19 — useDigitalWorldBootstrap 轮询中断动画

- **文件**：`packages/web/src/features/digital-world/hooks/useDigitalWorldBootstrap.ts:129-141`
- **描述**：`useWorldStatePolling` 每 5 秒 `setInterval` 拉取 `/v1/digital-world`，然后 `setData({ nodes, edges, being, ... })`。`WorldApp.setData` 在 id 集合未变时走"增量更新 being"分支，直接把 being **瞬移**到 `currentNodeId`，覆盖 `RouteAnimator` 正在进行的动画。

### C20 — WorldApp.redrawGrid 脆弱假设

- **文件**：`packages/web/src/features/digital-world/canvas/WorldApp.ts:317`
- **描述**：`redrawGrid()` 使用 `this.viewport.GetChildAt(0)` 移除旧 grid，**硬假设 grid 始终位于 viewport 子节点的索引 0**。但 `drawWorld()` 的 `addChild` 顺序为：grid → road → nodes → being → statusBubble。任何破坏该不变式的改动都会误删其他图层。

- **建议**：在 `WorldGrid` 上维护显式引用（如 `this.gridGraphics`）。

### C21 — ChatComposer 覆盖用户已选模型

- **文件**：`packages/web/src/pages/ChatPage/components/ChatComposer/ChatComposer.tsx:122-124`
- **描述**：
  ```ts
  const [model, setModel] = useState(defaultModelId);
  useEffect(() => { setModel(defaultModelId); }, [defaultModelId]);
  ```
  当用户在 Select 中手动切换到 `"seed"` 后，若 `useModels` 因任何原因重新渲染（如父组件 re-render），`useEffect` 会把用户的选择**强制重置**回 `defaultModelId`。
- **影响**：用户手动选择的模型被静默覆盖，后续消息以错误模型发送。
- **建议**：用 `userTouchedRef` 标记，仅在未手动设置时同步 `defaultModelId`。

### C22 — `evals` workspace 引用不存在的包

- **文件**：`/home/ubuntu/code/SunPilot/pnpm-workspace.yaml:4`；`/home/ubuntu/code/SunPilot/package.json:20-21`；`/home/ubuntu/code/SunPilot/.gitignore:22`
- **描述**：`pnpm-workspace.yaml` 把 `evals` 列为 workspace 包，根 `package.json` 定义 `eval:agent` / `eval:agent:real` 两个脚本 `--filter @sunpilot/evals`。但 `evals/` 目录实际不存在（glob 无结果），且 `.gitignore:22` 主动忽略 `evals/`。
- **影响**：`pnpm eval:agent` 必然报 "No projects matched the filter"，脚本是死脚本；workspace 声明一个不存在的包会污染 `pnpm -r` 的拓扑排序与 install；"被忽略的目录又是 workspace 成员"自相矛盾。
- **建议**：要么补齐 `evals/` 包并从 `.gitignore` 移除，要么从 workspace 与脚本中删除 `evals` 引用。

### C23 — 内存数据库 `transaction()` 不回滚

- **文件**：`packages/storage/src/testing/in-memory-database.context.ts:1129-1133`
- **描述**：`transaction()` 直接 `await work(this)`，不捕获错误回滚，也不支持嵌套事务。与 Postgres 实现行为严重不一致。
- **影响**：测试中即使 `work` 抛错，已写入的副作用也保留；测试可能"通过"但实际逻辑有缺陷；测试结果不能反映生产环境真实事务语义。
- **建议**：在 in-memory 实现中维护事务快照（深拷贝），抛错时恢复快照；至少在文档中明确"测试事务为 no-op"。

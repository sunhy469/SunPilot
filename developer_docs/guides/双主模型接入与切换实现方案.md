# SunPilot 双主模型接入与切换实现方案

更新时间：2026-06-17

## 目标

当前 SunPilot 只有一套 `SUNPILOT_LLM_*` 配置，实际是单主模型模式。目标是同时接入两个主模型：

- DP：现有 DeepSeek `deepseek-v4-flash`
- Seed：火山方舟豆包 `doubao-seed-2-0-lite-260428`

用户在聊天输入框的模型下拉里选择 `DP` 或 `Seed` 后，本次对话请求必须路由到对应模型，并保持现有 Agent Loop 的流式输出、工具调用、model_calls 记录、trace/debug 能力。

## 官方 API 依据

火山方舟提供 OpenAI 兼容接口。官方“Base URL 及鉴权”搜索索引显示 Chat Completions 地址为：

```text
https://ark.cn-beijing.volces.com/api/v3/chat/completions
Authorization: Bearer $ARK_API_KEY
Content-Type: application/json
```

官方“兼容 OpenAI SDK”文档说明方舟模型 API 尽可能兼容 OpenAI API，迁移时主要修改 `base_url`、`model`、`api_key`。

官方“对话(Chat) API / Responses API”文档目录包含 Chat API、Responses API、流式响应等入口；搜索索引也确认 `doubao-seed-2-0-lite-260428` 支持 `stream: true`。第三方模型 schema 页面同样显示该模型接受 ChatCompletion 形态的 `model/messages/stream` 参数，`stream=true` 时通过 SSE 返回增量 delta。

本次 SunPilot 的主对话链路已经基于 OpenAI-compatible Chat Completions 和 SSE 解析，因此 Seed 优先走现有 `OpenAICompatibleChatProvider`，不新增 Responses API adapter。Responses API 后续只在需要音频、视频、文件 `file_id` 等多模态能力时再接入。

参考链接：

- 火山方舟 Base URL 及鉴权：https://www.volcengine.com/docs/82379/1298459
- 火山方舟对话 Chat API：https://www.volcengine.com/docs/82379/1494384
- 火山方舟兼容 OpenAI SDK：https://www.volcengine.com/docs/82379/1330626
- 火山方舟 Responses API 流式响应：https://www.volcengine.com/docs/82379/1599499
- 豆包模型名与流式能力搜索证据：https://www.volcengine.com/docs/82379/1449737

## 现状判断

### 后端

现有 `packages/core/src/llm/openai-compatible.provider.ts` 已经满足 DP 和 Seed 的基础协议：

- POST `new URL("/chat/completions", baseUrl)`
- Bearer API Key 鉴权
- `stream: true`
- SSE `data:` 解析
- 从 `choices[].delta.content` 输出文本增量
- 兼容 `tools` 和 `tool_choice`

现有 `packages/core/src/agent-kernel/model-router.ts` 已经有 `ModelRouter`、`ModelPurpose`、fallback、`model_calls` 记录能力。但 `packages/daemon/src/composition-root.ts` 现在仍使用：

```ts
createSingleModelRouter(deps.llmProvider, undefined, deps.database.modelCalls)
```

因此所有 purpose 都被固定路由到同一个 provider。

### 前端

`packages/web/src/pages/ChatPage/components/ChatComposer.tsx` 已有模型下拉，但目前是假的 UI 状态：

```ts
const MODEL_OPTIONS = [
  { value: "claude-sonnet-4-6", label: "Sonnet 4.6" },
  { value: "claude-opus-4-8", label: "Opus 4.8" },
  { value: "claude-haiku-4-5", label: "Haiku 4.5" },
];
const [model, setModel] = useState("claude-sonnet-4-6");
```

`onSend` 只传 `text/attachments/permissionMode`，没有传 model。`packages/web/src/features/chat/ws.ts` 也没有把模型选择写进 JSON-RPC params。

另外，`sendChatMessage()` 目前声明了 `permissionMode`，但实际 JSON-RPC params 里也没传 `permissionMode`。本次加模型字段时应一起修正，否则前端选择仍然不会到 daemon。

## 设计原则

1. `modelId` 是用户请求级别的选择，不应写死为全局配置。
2. DP 和 Seed 都是主模型，不是 fallback。fallback 只用于主模型失败后的降级策略。
3. 所有 Agent Loop 模型调用必须继续通过 `ModelRouter`，避免绕开 `model_calls` 和 trace。
4. Seed 先复用 OpenAI-compatible Chat Completions；不在本次方案里切 Responses API。
5. 前端下拉值必须从协议透传到 `AgentLoopInput`，不能只存在 React 本地状态。

## 配置方案

保留现有 `SUNPILOT_LLM_*` 作为 DP 默认配置，同时新增 Seed 专用配置：

```bash
# DP / DeepSeek
export SUNPILOT_DP_LLM_BASE_URL="https://api.deepseek.com"
export SUNPILOT_DP_LLM_MODEL="deepseek-v4-flash"
export SUNPILOT_DP_LLM_API_KEY="..."

# Seed / Volcengine Ark
export SUNPILOT_SEED_LLM_BASE_URL="https://ark.cn-beijing.volces.com/api/v3"
export SUNPILOT_SEED_LLM_MODEL="doubao-seed-2-0-lite-260428"
export SUNPILOT_SEED_LLM_API_KEY="..."

# 默认 UI 和无显式选择时使用的模型
export SUNPILOT_DEFAULT_CHAT_MODEL="dp"
```

兼容策略：

- 若 `SUNPILOT_DP_LLM_*` 缺失，则回退读取旧的 `SUNPILOT_LLM_*`。
- 若 `SUNPILOT_SEED_LLM_API_KEY` 缺失，后端仍可启动，但 `/v1/models` 或聊天请求应标记 Seed unavailable。
- 不要复用 `SUNPILOT_LLM_API_KEY` 给 Seed，避免不同供应商 key 混用。

## 协议变更

新增受控枚举：

```ts
export type ChatModelId = "dp" | "seed";
```

需要更新：

- `packages/protocol/src/agent-commands.ts`
  - `chatSendSchema` 增加 `modelId: z.enum(["dp", "seed"]).optional()`
  - `ChatSendParams` 增加 `modelId?: ChatModelId`
- `packages/web/src/features/chat/types.ts`
  - `ChatSendParams` 增加 `modelId?: "dp" | "seed"`
- `packages/web/src/features/chat/ws.ts`
  - JSON-RPC params 透传 `modelId`
  - 同时补传当前遗漏的 `permissionMode`
- `packages/api/src/ws/json-rpc-router.ts`
  - `agent.startChatCommand()` 输入增加 `modelId: params.modelId`
- `packages/core/src/agent/agent.service.ts`
  - `startChatCommand()`、`handleChatCommand()` 输入增加 `modelId?: ChatModelId`
  - `AgentLoopInput` 增加 `modelId?: ChatModelId`
  - run input 持久化时保留 `modelId`，用于 retry/resume 复用原模型

## 后端实现方案

### 1. 扩展 LLM 配置读取

在 `packages/core/src/llm/llm.config.ts` 或 daemon 侧新增双模型配置解析：

```ts
interface NamedLlmProviderConfig {
  id: "dp" | "seed";
  label: string;
  provider: "deepseek" | "volcengine-ark";
  baseUrl: string;
  model: string;
  apiKey: string;
  available: boolean;
}
```

构造两个 `OpenAICompatibleChatProvider`：

```ts
const dpProvider = new OpenAICompatibleChatProvider({
  apiKey: dp.apiKey,
  baseUrl: dp.baseUrl,
  model: dp.model,
});

const seedProvider = new OpenAICompatibleChatProvider({
  apiKey: seed.apiKey,
  baseUrl: seed.baseUrl,
  model: seed.model,
});
```

建议给 provider 增加可选 `id`：

```ts
new OpenAICompatibleChatProvider({
  id: "llm.deepseek",
  apiKey,
  baseUrl,
  model,
});

new OpenAICompatibleChatProvider({
  id: "llm.volcengine-ark",
  apiKey,
  baseUrl,
  model,
});
```

这样 `model_calls.provider` 能区分供应商，而不是都写 `llm.openai-compatible`。

### 2. 让 ModelRouter 支持请求级强制模型

现在 `ModelRouter.streamChat(purpose, request)` 只看 purpose。需要给 `ChatCompletionRequest` 增加：

```ts
modelId?: "dp" | "seed";
```

`ModelConfig` 增加：

```ts
id?: "dp" | "seed";
label?: string;
```

`ModelRouter.streamChat()` 选路顺序改为：

1. 如果 `request.modelId` 存在，优先选 `config.id === request.modelId` 且支持该 purpose 的 route。
2. 如果找不到该模型 route，抛出可读错误：`Model "seed" is not configured for response_composition`。
3. 如果没传 `modelId`，使用 `SUNPILOT_DEFAULT_CHAT_MODEL` 对应 route。
4. 当前 route 失败时，只在明确配置 fallback 的情况下 fallback；不要自动从 DP 切到 Seed，除非产品接受“用户选 Seed 但实际 DP 回答”。

所有 purpose 都可以先给 DP 和 Seed 配同一组：

```ts
const allPurposes: ModelPurpose[] = [
  "intent_classification",
  "tool_argument_generation",
  "reflection",
  "response_composition",
  "summary_compression",
  "planning",
  "replanning",
  "clarification",
];
```

### 3. 在 Agent Loop 里传递 modelId

`AgentLoopInput` 携带 `modelId` 后，以下调用都需要把它写进 model request：

- `ResponseComposer.composeDirect()`
- `ResponseComposer.composeFromObservation()`
- `ToolDecisionEngine.executeStreaming()` / `streamLlmTurn()`
- `BasicReflectionEngine`
- `ToolArgumentBuilder`
- `ToolDecisionEngine.decide()` 中所有 `llm.streamChat()`
- planning / replanning 的 LLM 调用

最小实现可以在 `createPurposeProvider(purpose)` 的 `streamChat(request)` 中合并当前 run 的 modelId，但更清晰的方式是把 `modelId` 放入每个 `ChatCompletionRequest`。这样 `model_calls.metadata` 也能记录：

```ts
metadata: {
  ...request.metadata,
  selectedModelId: request.modelId,
}
```

### 4. 保持 Seed 流式输出

Seed 仍走 `OpenAICompatibleChatProvider.streamChat()`：

```json
{
  "model": "doubao-seed-2-0-lite-260428",
  "messages": [{ "role": "user", "content": "hello" }],
  "stream": true
}
```

现有 SSE 解析只依赖 `data: ...` 和 `choices[].delta.content`，与火山方舟 OpenAI-compatible Chat Completions 对齐。需要补充一个 provider 单测，mock 火山返回：

```text
data: {"choices":[{"delta":{"content":"你"}}]}

data: {"choices":[{"delta":{"content":"好"}}]}

data: [DONE]
```

断言最终 yield `你`、`好`。

## 前端实现方案

### 1. 替换模型下拉

`ChatComposer.tsx` 改为真实选项：

```ts
const MODEL_OPTIONS = [
  { value: "dp", label: "DP" },
  { value: "seed", label: "Seed" },
];
```

默认值读取后端 `/v1/models` 或先使用 `"dp"`。

`onSend` 签名改为：

```ts
onSend(
  text: string,
  attachments?: AttachmentRef[],
  permissionMode?: "ask" | "auto" | "full",
  modelId?: "dp" | "seed",
): void;
```

`handleSend()` 调用：

```ts
onSend(text, attachments, permission, model as "dp" | "seed");
```

### 2. WebSocket 透传

`useChat.send()` 签名增加 `modelId`，调用 `sendChatMessage()` 时带上：

```ts
sendChatMessage(socket, {
  conversationId,
  message: text,
  mode: "agent",
  permissionMode: permissionMode ?? "auto",
  modelId: modelId ?? "dp",
  attachments,
});
```

`sendChatMessage()` JSON-RPC params 修正为：

```ts
params: {
  conversationId: params.conversationId,
  message: params.message,
  mode: params.mode ?? "agent",
  permissionMode: params.permissionMode,
  modelId: params.modelId,
  clientRequestId: params.clientRequestId,
  attachments: params.attachments,
}
```

### 3. 模型可用性 API

建议新增：

```http
GET /v1/models
```

返回：

```json
{
  "defaultModelId": "dp",
  "items": [
    {
      "id": "dp",
      "label": "DP",
      "provider": "deepseek",
      "model": "deepseek-v4-flash",
      "available": true
    },
    {
      "id": "seed",
      "label": "Seed",
      "provider": "volcengine-ark",
      "model": "doubao-seed-2-0-lite-260428",
      "available": true
    }
  ]
}
```

前端下拉禁用 unavailable 模型，并在 tooltip 或二级文本显示“未配置 API Key”。

## 测试与验收

### 单元测试

- `OpenAICompatibleChatProvider`
  - DeepSeek URL `https://api.deepseek.com` 拼出 `/chat/completions`
  - Seed URL `https://ark.cn-beijing.volces.com/api/v3` 拼出 `/chat/completions`
  - SSE delta 正确流式 yield
- `ModelRouter`
  - `modelId=dp` 时调用 DP provider
  - `modelId=seed` 时调用 Seed provider
  - 未配置 Seed 时返回可读错误
  - `model_calls` 记录 provider/model/selectedModelId
- `chatSendSchema`
  - 接受 `modelId=dp|seed`
  - 拒绝未知模型值
- `sendChatMessage`
  - JSON-RPC params 包含 `modelId`
  - JSON-RPC params 包含 `permissionMode`

### 集成测试

- WebSocket `chat.send` 携带 `modelId=seed`，后端 `agent.model.started` 事件中的 `model` 为 `doubao-seed-2-0-lite-260428`。
- `agent.response.delta` 能持续返回 Seed 文本增量。
- `GET /v1/runs/:id/model-calls` 能看到 Seed 的 model call。
- UI 下拉切换 DP/Seed 后，相邻两次对话分别记录不同模型。

## 分阶段落地

### Phase 1：协议和 UI 真实透传

- 替换 `MODEL_OPTIONS` 为 DP/Seed。
- `ChatComposer -> useChat -> sendChatMessage -> JSON-RPC` 透传 `modelId`。
- 同时修复 `permissionMode` 未透传。
- `chatSendSchema` 接受 `modelId`。

### Phase 2：双 provider 和请求级路由

- 新增 DP/Seed env 解析。
- 构造两个 `OpenAICompatibleChatProvider`。
- 扩展 `ModelRouter` 支持 `request.modelId`。
- `AgentLoopInput`、`ChatCompletionRequest` 透传 `modelId`。

### Phase 3：可观测性和配置 API

- `model_calls.metadata.selectedModelId`。
- `agent.model.started` 事件显示真实 provider/model。
- 新增 `/v1/models` 给前端读取可用模型。

### Phase 4：Seed 真实流式验收

- 用火山方舟真实 key 做 WebSocket smoke。
- 确认 Seed 返回 `agent.response.delta`。
- 确认 Debug 面板、trace、model_calls 与 DP 并列可查。

## 风险点

- Seed 的 Chat Completions 是否支持当前 native `tools/tool_choice` 需要实测。如果不支持，应在 `ModelConfig.capabilities` 里标记 `toolCalling: false`，工具选择/参数生成 purpose 默认仍走 DP，只有 `response_composition` 走 Seed。
- 用户选择 Seed 但内部 intent/planning 仍走 DP，会造成“主模型”语义不清。建议第一版先让所有 purpose 都跟随用户选择，若 Seed tool calling 不稳定，再显式拆成“回答模型”和“工具规划模型”。
- 如果后端自动 fallback 到另一个主模型，UI 可能显示 Seed 但实际 DP 回答。建议默认不跨主模型 fallback，只在 Debug 中记录失败原因。
- 现有“空白 assistant card”问题与模型接入无直接关系，但双模型 smoke 时会更容易暴露。若同一 run 出现两个 `messageId`，应先按 run 绑定 streaming assistant message。

## 结论

Seed 接入不需要新建一套流式协议；按官方 OpenAI-compatible Chat Completions 接入即可。真正需要改的是 SunPilot 当前“单 provider 全局配置”和“前端模型选择未透传”的两处结构。完成后，DP 和 Seed 都作为主模型出现在同一个对话框模型下拉中，每次 chat.send 按用户选择路由，流式输出和 debug/model_calls 继续沿用现有 Agent Loop 能力。

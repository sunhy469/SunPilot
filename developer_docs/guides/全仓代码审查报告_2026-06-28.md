# SunPilot 全仓代码审查报告

审查日期：2026-06-28

代码基线：`main` / `6ca9596`

审查范围：`packages/*/src`、`tests/integration`、根级工程配置、PostgreSQL migrations、前端生产构建及生产依赖树。

审查约束：本次只审查与写文档，没有修改任何产品代码。

## 1. 结论先行

SunPilot 已经具备较完整的 Agent Runtime 骨架：包边界清楚，Agent Loop、内容块流式输出、工具调用、审批、PostgreSQL 持久化、记忆、追踪、Web Chat 与 Digital World 都不是空壳。但当前版本还不能按“生产安全、行为闭环、门禁可信”验收。

本轮确认的最高优先级问题不是代码风格，而是以下真实运行风险：

1. HTTP 本地 token 鉴权可被无 `Origin` 请求绕过；
2. `PromptInjectionDetector`、`ToolSandbox`、`TaskScopedPermissionManager` 虽已实例化，却没有进入实际工具执行判定；
3. 第三方 Skill 仍通过动态 `import()` 在 daemon 进程内执行，可直接绕过 SDK 权限接口；
4. Digital World 的异步任务续跑、字段清空和 artifact 桥接存在多处确定性错误；
5. 迁移目录已有 `022`，但迁移执行器只运行到 `021`，新数据库与当前开发库会产生结构分叉；
6. 全仓类型检查失败，但 `pnpm build` 仍会成功，当前构建门禁会放过前端类型错误；
7. 测试会继承真实 Seed 密钥并访问外部模型，造成失败、成本和数据外发风险。

审查结果按严重度汇总：

| 严重度 | 数量 | 含义 |
| --- | ---: | --- |
| P0 | 3 | 安全边界失效，发布前必须处理 |
| P1 | 14 | 核心功能、数据一致性或可靠性缺陷，应进入最近修复批次 |
| P2 | 9 | 中期工程治理、性能和契约问题 |

## 2. 审查方法与覆盖面

本次不是只抽查 Agent Loop，而是采用以下组合方式：

- 盘点 10 个运行包、321 个非测试 TypeScript/TSX 源文件、60 个测试文件、22 个 SQL migration；
- 阅读 daemon 装配根、HTTP/WS 入口、AgentService、AgentLoop、native tool loop、安全策略、SkillRunner、PostgreSQL repositories、Digital World 与 Web 状态链路；
- 对事件、模型、权限、附件、审批、任务、artifact、memory、迁移和 CLI 进程控制做跨包追踪；
- 执行 `typecheck`、`lint`、`test`、`build` 和生产依赖安全审计；
- 只读查询本机 PostgreSQL 的 migration/constraint 实况，用于识别“当前开发库可用但新安装会缺失”的问题；
- 对每个高优先级问题同时核对生产代码路径与现有测试覆盖，避免把旧文档结论当作当前事实。

代码规模约 71,236 行 TypeScript/TSX，其中 `core` 约 35,024 行、`web` 约 18,635 行、`storage` 约 6,202 行、`daemon` 约 5,314 行。

## 3. P0 问题

### P0-01：无 Origin 的 HTTP 请求可绕过本地 bearer token

证据：

- `packages/daemon/src/server.ts:287-299`
- `isAllowedLocalOrigin()` 会接受 `localhost` / `127.0.0.1` 且端口在允许集合中的 origin；
- token 校验失败后，代码把缺失的 `Origin` 合成为 ``https://${request.hostname}``；
- 对 `curl`、脚本或同机恶意进程，`request.hostname` 通常就是 `127.0.0.1` 或 `localhost`，合成值会通过 origin 检查。

实际影响：

- 注释宣称“无 Origin 的非浏览器客户端必须提供 token”，实际行为相反；
- 任意能访问 daemon 端口的无 Origin 客户端可以调用对话、配置、审批、Skill、memory、artifact 等非豁免 API；
- 若未来通过配置或代码将 daemon 绑定到非 loopback 地址，伪造 `Host: localhost` 还会扩大风险。

建议：

1. token 启用时，无有效 bearer token 就直接拒绝 API 请求；不要用 Origin 代替认证；
2. 如果必须支持同源 SPA，采用一次性 bootstrap/session、HttpOnly cookie 或明确的 CSRF 设计；
3. `Origin` 只作为额外边界，不应成为身份凭证；
4. 增加真实 auth 测试：无 Origin、伪造 Host、恶意 Origin、合法 token、WebSocket query token、认证关闭模式。

验收标准：token 开启时，无 token 的 `curl`/脚本请求返回 401；只有健康检查保持匿名。

### P0-02：三个安全模块已装配但没有执行

证据：

- `packages/daemon/src/composition-root.ts:719-739` 创建 `PromptInjectionDetector`、`ToolSandbox`、`TaskScopedPermissionManager`；
- `packages/daemon/src/composition-root.ts:750-772` 只把它们放入 `AgentLoopEngine` 依赖；
- `packages/core/src/agent-kernel/agent-loop-engine.ts` 中 `toolSandbox` 和 `scopedPermissionManager` 没有校验调用，`grantsByRun` 只被清理；
- 唯一的 `injectionDetector.detect()` 位于 native tool loop，但 `ToolDecisionEngine` 在 `composition-root.ts:676-689` 创建时没有收到 `injectionDetector`，所以该分支在生产装配下始终不运行；
- `agent.safety.injection_detected`、`agent.safety.sandbox_denied`、`agent.safety.scope_reauth_required` 有协议类型，但当前主路径不会产生对应防护结果。

实际影响：

- `SUNPILOT_SANDBOX_MODE=strict` 只改变启动日志，不改变工具执行结果；
- 工具返回内容不会按当前设计做 prompt injection 阻断；
- 参数变化、同 run 授权范围和高风险重新确认没有被 task-scoped manager 强制执行；
- 现有架构文档把这些能力描述为“运行路径已接入”，与源码事实不一致。

建议：

1. 把安全检查放到唯一、不可绕过的 execution boundary；
2. 让 native tool call、审批续跑、传统 orchestrator 三条路径复用同一 `SafetyDecision`；
3. 先校验 sandbox/注入/参数作用域，再创建 tool started 事件和执行 step；
4. 对每个安全模块增加“装配测试 + 拒绝测试”，不能只测类本身。

验收标准：strict 模式下构造越界文件、危险 shell、未允许网络和注入型工具结果，均应被阻断并产生持久化安全事件。

### P0-03：第三方 Skill 可在 daemon 进程内任意执行代码

证据：

- `packages/skill-runner/src/runner.ts:148` 直接动态导入 Skill entry；
- 模块顶层代码在 manifest、capability 和 SDK handler 校验之前已经执行；
- Node 模块可以自行导入 `node:fs`、`node:child_process`、读取 `process.env` 或直接调用 `fetch`，无需经过 `ctx.files/http/secrets`；
- runner 的超时是 `Promise.race()`（`runner.ts:330-351`），Skill 若忽略 `AbortSignal`，超时后仍可在后台继续运行。

实际影响：

- manifest allowlist 只是对守规矩 Skill 的 SDK 约束，不是安全沙箱；
- 任意已安装/被替换的 Skill 等价于 daemon 进程权限代码；
- 超时后释放并发槽位并不能终止恶意或失控代码，可能形成资源泄漏和并发失控。

建议：

1. 不可信 Skill 必须迁移到 child process、Worker 隔离或容器；
2. 主进程仅通过结构化 IPC 提供受控文件、网络、secret 和 artifact API；
3. 超时后终止执行单元，而不是只 abort 协作信号；
4. 增加包 hash/签名/来源锁定，处理 symlink 逃逸与重定向后 SSRF；
5. 在隔离完成前，把 Skill 明确标记为“本机可信代码插件”。

## 4. P1 问题

### P1-01：全仓类型检查失败，生产构建却为绿色

执行 `pnpm typecheck` 得到 3 个错误：

1. `packages/web/src/pages/ChatPage/hooks/chat/chat-state.ts:324-325` 的 `../../../rich-cards/types` 相对路径少了一层；
2. `packages/web/src/pages/ChatPage/hooks/useChat.ts:528` 使用了未导入的 `ChatSendAckResult`。

`packages/web` 的 `build` 仅运行 `vite build`，esbuild 会擦除这些类型引用，因此根级 `pnpm build` 仍成功。这会让 CI/发布误判“编译通过”。

建议：把 `tsc --noEmit` 作为 `packages/web` build 前置步骤，并在根 CI 中强制执行 `typecheck && lint && test && build`。

### P1-02：附件-only 消息在 UI 可发送，但协议层必然拒绝

证据：

- `packages/web/src/pages/ChatPage/hooks/useChat.ts:819-824` 明确允许“文本为空但有附件”；
- `packages/protocol/src/agent-commands.ts:13` 要求 `message` 至少 1 个字符；
- HTTP `parseAgentChatRequest()` 同样拒绝空文本。

影响：用户只上传图片并发送时，UI 先插入 optimistic 用户消息和 assistant placeholder，随后收到 JSON-RPC invalid params；图片理解/搜图的自然入口不可用。

建议：统一成“`message.trim()` 非空或 `attachments.length > 0` 至少满足一个”的 schema，并为 WS、REST、AgentService、UI 添加同一组契约测试。

### P1-03：模型配置、默认路由和测试环境互相冲突

该问题包含三个相互放大的子问题：

1. `createDefaultLlmProvider()` 只认 `SUNPILOT_LLM_API_KEY` / `DEEPSEEK_API_KEY`，而 composition root 还定义 `SUNPILOT_DP_LLM_API_KEY`；只配 DP 专用 key 仍在 agent 初始化前失败；
2. `/v1/models` 和 Web fallback 都固定默认 `seed`，即使 Seed `available: false`，常规单 DP 部署会默认选择不可用模型；
3. composition root 只要从环境读到 Seed key，就把 Seed 设为优先级 0。即使测试显式注入 fake `llmProvider`，未显式 `modelId` 的调用仍会访问真实 Seed。

本轮实测：

- 直接 `pnpm test` 时，`composition-root.test.ts` 访问了真实 Seed provider，并有 3 个 daemon 用例失败；
- 使用 `env -u SUNPILOT_SEED_LLM_API_KEY pnpm test` 后，全套测试通过。

影响：测试不具备 hermetic 性，会产生外部费用、发送测试 prompt/上下文，并因真实模型输出变化而随机失败。

建议：

- 建立唯一 `resolveModelConfig()`；
- API 默认值必须从“实际可用 route”推导；
- 测试装配显式禁用环境 provider，或依赖注入一旦存在就完全覆盖环境；
- CI 清空全部模型/OSS secret，并用 fake provider；真实模型 eval 放到独立、显式任务。

### P1-04：WebSocket fast-ack 的同步准备阶段缺少失败清理

`AgentService.startChatCommand()` 在 `queueMicrotask()` 之前依次保留幂等键、创建 abort controller、订阅两个 event bus、创建 conversation/run/message。只有进入后台任务后才有清理逻辑。

确定性问题：

- 图片引用校验发生在幂等键 reserve 之后；校验抛错会留下最长 24 小时的 `processing` 记录；
- conversation/run/message 任一步失败会泄漏 abort entry 和订阅；
- retry 使用同一 `clientRequestId` 会得到 idempotency conflict，而不是正常重试。

建议：把所有 fast-ack 准备步骤放进 `try/catch/finally`；校验必须早于 reserve；失败时回滚/标记幂等失败并撤销订阅和 abort entry。

### P1-05：assistant 消息写库失败会被静默当成成功

证据：

- `packages/daemon/src/composition-root.ts:157-172` 的 `saveMessage()` 捕获并吞掉所有数据库异常；
- `AssistantMessageStream.complete()` 自身也把 save 失败降级为日志，然后继续发 `agent.message.completed`；
- run 随后可被标记为 `completed`。

影响：前端看到完整回复和成功 run，但刷新后 assistant 消息消失；事件、run 与 message 不再满足持久化一致性。

建议：embedding 失败可以 best-effort，消息主体写入失败不能吞掉。应重试、标记 run failed/interrupted，或将最终消息与 run completion 放入同一事务/outbox。

### P1-06：审批路径在动态风险和多工具场景下不闭环

两个问题：

- native executor 在参数生成后调用 `PermissionPolicy`。如果这时才判定需要审批，`streaming-tool-call-executor.ts:348-370` 只返回“需要审批”的失败文本，不创建 approval，也不把 run 置为 `waiting_approval`；
- 初始 decision 含多个高风险 tool call 时，`approval-flow.ts:104-120` 为每个 tool 单独创建 approval，但 `approve()` / `resumeApprovedTool()` 每次只执行一个 requested action 并可能完成整个 run。剩余 approval 会指向已完成 run。

建议：动态风险结果必须返回结构化 `require_approval`，由统一 approval coordinator 持久化；多工具应采用批次审批或维护明确的 pending tool queue，并在全部处理后才结束 run。

### P1-07：Digital Being 的“清空字段”全部是 no-op

业务代码大量使用以下 patch：

- `currentTaskId: undefined`
- `currentActionId: undefined`
- `targetNodeId: undefined`
- `sleepReason: undefined`

但 `PostgresDigitalBeingRepository.update()` 只更新 `value !== undefined` 的字段。因此任务结束、移动完成、唤醒和失败恢复都不会把数据库列设为 `NULL`。

影响：Digital Being 会长期残留旧 task/action/target/sleepReason，UI 和后续调度读取到伪状态。内存实现若用对象覆盖可能没有同样行为，进一步扩大测试与生产差异。

建议：patch 类型引入显式 `null` 清空语义，并在 repository 中将 `null` 写入 SQL；为每个生命周期动作做 PostgreSQL 集成测试。

### P1-08：Digital World 异步任务续跑会越过第二个 Agent Run

首次 `executeTask()` 遇到 `waiting_agent` 会停止后续 action，这是正确的。但 `onAgentRunCompleted()` 恢复剩余 actions 时调用 `executeAction()` 后忽略返回值。

对 `make_and_publish_video`：

1. 第一个 `work_on:make_video` 等待 Agent；
2. 完成回调继续执行到第二个 `work_on:publish_to_tiktok`，启动新的 Agent Run；
3. 回调不再暂停，继续 move home、sleep，并把 task 标记 completed；
4. 第二个 Agent Run 仍在后台运行。

建议：恢复循环和初始循环共用同一个状态机；遇到任意后续 `waiting_agent` 都必须立即返回，并持久化恢复游标。

### P1-09：Agent artifact 到 Digital World artifact 的 payload 形状不一致

Agent Loop 的 `agent.run.completed.payload.artifacts` 当前发送 artifact ID 字符串数组；daemon 的 Digital World bridge 却把每个元素按 `{type,title,name,uri}` 对象读取。

影响：创建出的 WorldArtifact 会退化为 `type: "unknown"`、`title: "产物"`、无 URI，真实产物信息丢失。

建议：协议明确 payload 是 ID 数组还是对象数组。若是 ID，应在 bridge 中通过 artifact repository 批量加载；不要对字符串做对象强制断言。

### P1-10：Digital World 的任务 claim 并不原子

`claimTask()` 虽在 transaction 中执行，但仍是普通 `SELECT` 后 `UPDATE`，没有 `SELECT ... FOR UPDATE`，也没有条件更新 `UPDATE ... WHERE status='queued' RETURNING ...`。

两个并发事务可同时读取 `queued`，随后都改为 `running` 并重复创建/执行 actions。

建议：使用单条条件更新，或在事务内行锁；为并发 claim 增加真实 PostgreSQL 测试。

### P1-11：迁移执行器漏掉仓库中的 022 migration

`packages/storage/src/migrations/022_digital_world_foreign_keys.sql` 已存在，但 `postgres.migrations.ts:7-29` 的白名单只到 `021_memory_quality_relations.sql`。

本机数据库只读检查显示已经登记 `022` 和历史 `023`，所以当前开发库有相关外键/索引；但干净数据库按当前代码初始化时不会执行 022。这是典型的“老库正常、新安装缺约束”。仓库中也没有当前数据库已登记的 023 文件。

建议：迁移清单与文件目录建立自动一致性校验；禁止数据库出现仓库不存在的版本；在空数据库上执行完整 migration smoke test 并校验关键 FK/index。

### P1-12：CLI 可能因陈旧 PID 文件终止无关进程

`packages/launcher/src/index.ts:138-169` 读取 pid file 后直接 `SIGTERM`，不验证该 PID 是否仍属于 SunPilot；PID 被系统复用时会终止无关进程。fallback 又按命令行 substring 扫描，仍可能误匹配。

建议：PID 文件同时记录进程启动时间和随机实例 token；停止前比对 `/proc/<pid>/cmdline`、启动时间及 daemon health identity；匹配失败只删除 stale pid，不发送信号。

### P1-13：Memory hybrid search 会让 NULL embedding 排到最前，并限制语义召回

`postgres.memory.repository.ts` 中：

- `semanticSql` 对无 embedding 记录返回 `NULL`；
- `ORDER BY score DESC` 在 PostgreSQL 中默认把 NULL 放前面；
- query + embedding 同时存在时仍用 ILIKE 做硬前置过滤，纯语义相关但没有关键词重合的记忆永远进不了候选集。

影响：未嵌入记录可能挤占 top-k；所谓 hybrid recall 实际退化成“词法命中集合内再向量排序”。

建议：对 semantic score 使用 `COALESCE` 或 `NULLS LAST`；采用 lexical/vector 两路候选 union 后再融合排序；增加同义改写和 NULL embedding 的 PostgreSQL 回归测试。

### P1-14：上传大小限制可通过省略 sizeBytes 绕过

`uploadPresignBodySchema` 把 `sizeBytes` 定义为 optional，`OssClient` 也只在该字段存在时校验。调用者省略大小即可获得 PUT URL；当前签名没有绑定 content-length 上限。

建议：presign 必须要求可信 `sizeBytes`，并在 OSS policy/回调层验证真实对象大小；文件名 public URL 也应逐段 URL encode，避免 `#`、`%`、空格导致不可用链接。

## 5. P2 问题

### P2-01：生产 Digital World 失败时会把 mock 数据伪装成真实数据

hook 初始 state 就是 mock。生产环境请求失败或返回空数据时，代码只把 `loaded=true, isMock=false` 合并到旧 state，没有清空 mock nodes/beings。结果是生产页面显示 mock 世界，却不显示 `DEV MOCK` 标识。

建议：生产失败态使用空数组和显式 error；mock 只在 DEV 分支创建，不能作为生产初始 state。

### P2-02：多数字生命选择后，唤醒/休眠仍操作 primary being

面板已使用 `activeBeing`，但 `DigitalWorld.tsx:77-90` 的 `handleWakeSleep()` 仍引用 `being`。选中其他数字生命后，dock 的唤醒/休眠会误操作第一个 being。

### P2-03：配置文件公开了多个不生效选项

`config.json` 中的 `server.port`、`security.requireLocalToken`、`skills.directories`、`skills.autoReload` 可以读写，但 daemon 的端口、token 开关和 SkillRegistry 并不从这些字段装配。API 返回成功会让使用者误以为配置已生效。

建议：要么接入运行时，要么从可写 schema 移除并标记只读/未支持。

### P2-04：Memory retry 发出协议外事件

`MemoryRetryWrapper` 通过 `as any` 发出 `agent.memory.write_failed`，但该 method 不在 `AGENT_EVENT_TYPES`；前端和协议消费者不会稳定识别。应统一使用 `agent.error` + memory category，或正式扩展协议并补 schema/consumer。

### P2-05：错误响应泄露内部异常文本

HTTP 全局 500、`/v1/chat` 和 JSON-RPC internal error 都直接返回 `Error.message`。数据库 SQL、上游 LLM response body、本机路径等信息可能暴露给客户端。

建议：客户端返回稳定 error code/request id，详细异常只写结构化日志并做 secret redaction。

### P2-06：限流器会永久保留访问过的 IP key

`SlidingWindowRateLimiter` 会裁剪单个 bucket 的时间戳，但从不删除空 bucket。若 daemon 对外暴露或经过大量伪造源地址，Map 会持续增长。

建议：空 bucket 删除、定期 sweep，并限制全局 key 数；生产部署优先使用成熟 rate-limit 插件/反向代理。

### P2-07：Embedding dimensions 配置与 provider/schema 不一致

环境变量允许修改 `SUNPILOT_EMBEDDING_DIMENSIONS`，但默认 embedding provider 没有把该值传入，数据库列固定为 `vector(1536)`。非 1536 设置会造成 service 声明、provider 返回和数据库 schema 三方不一致。

建议：若只支持 1536，就把 env 收紧为 literal 1536；若支持多维度，需要 migration/model config 的原子切换策略。

### P2-08：RichText 链接缺少 URL scheme allowlist

`RichTextRenderer` 和 `linkify()` 会直接把结构化 `href` 或 markdown link 写入 `<a href>`。应只允许 `https:`, `http:`, `mailto:` 等明确 scheme，拒绝 `javascript:`、`data:` 和畸形 URL；媒体 URL 同样需要策略。

### P2-09：Digital World 完成回调为 O(beings × actions) 扫描

`findActionByRunId()` 逐个读取 being，再逐个拉 action。当前数据库曾有 `idx_world_actions_agent_run`，但对应 023 migration 不在仓库。应增加 repository 的 `findByAgentRunId()` 和受版本控制的索引 migration。

测试通过不能覆盖本报告中的 Digital World 风险：`packages/platform` 使用 `--passWithNoTests`，实际没有测试文件；集成测试只有一个 WebSocket happy path，并显式关闭 token auth。安全、模型默认值、真实 PostgreSQL lifecycle、多工具审批和 clean-database migration 都没有端到端门禁。

Web 测试虽然通过，但 stderr 已出现：

- RecentConversations 缺少稳定 React key；

## 9. 推荐修复顺序

### 第一批：发布阻断项

1. 修复 HTTP/WS 认证模型并补 auth 攻击面测试；
2. 把 prompt injection、sandbox、task-scoped permission 接到不可绕过的工具执行入口；
3. 明确 Skill 仅可信，或完成进程级隔离；
4. 修复 Web typecheck，并让 build/CI 必须运行 tsc；
5. 隔离测试环境中的全部真实 provider secret。

### 第二批：核心状态一致性

1. 修复 assistant message 持久化失败语义；
2. 统一动态风险审批和多工具审批队列；
3. 修复 Digital Being nullable patch、异步 action 恢复和 artifact payload；
4. 把 task claim 改为条件更新/行锁；
5. 修复 migration 022/023 的仓库与数据库版本一致性。

### 第三批：契约与检索质量

1. 统一 attachment-only、REST/WS chat 和模型默认值；
2. 修复 memory hybrid candidate/NULL score；
3. 强制上传大小并治理 public URL；
4. 收敛 config schema、embedding dimensions 和事件词汇；
5. 升级有漏洞的生产依赖。

### 第四批：测试与可维护性

1. 为 `platform`、`protocol`、`api` 增加包级测试；
2. 增加 clean PostgreSQL migration、并发 task、auth、审批、Skill sandbox 集成测试；
3. 将 `composition-root.ts` 拆为 model/context/tool/safety/persistence 工厂，并对每个工厂做 wiring test；
4. 建立真实模型 eval，但与默认单元/集成测试严格隔离；
5. 清理 React warnings、deprecated API 和 bundle 体积。

## 10. 最终判断

架构方向仍然成立，但当前代码的“文件存在”和“生产能力已生效”之间还有明显落差。最典型的是安全模块：类型、实现、配置和启动日志都存在，却没有成为真实执行门。Digital World 也已经跨过 UI mock 阶段，但状态机和 PostgreSQL patch 语义尚未形成可靠闭环。

因此，本轮建议不是继续增加新模块，而是先完成三项收口：

1. 把认证、工具安全和 Skill 隔离变成真实边界；
2. 把消息、审批、任务和 migration 变成可事务验证的状态机；
3. 把 typecheck、无外网测试、空库 migration 和安全回归变成发布门禁。

完成这些工作后，SunPilot 才适合从“功能完整的 Agent Runtime 开发版”进入“可持续发布的生产候选版”。

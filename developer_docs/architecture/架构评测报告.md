# SunPilot 当前架构评测报告

生成日期：2026-06-09　|　更新日期：2026-06-11（Workflow 清理后复审）

评测对象：当前工作区实际代码，覆盖 `packages/protocol`、`packages/core`、`packages/api`、`packages/daemon`、`packages/storage`、`packages/skill-runner`、`packages/skill-sdk`、`packages/web`、`packages/launcher`、`tests/integration`。`packages/workflow` 已在重构中移除。

本报告已按当前代码重新评测，旧版报告中关于 `SunPilotRuntime`、旧 runtime 主路径、`POST /v1/runs` 返回 410 等结论已过期，不再采用。

## 1. 总体结论

SunPilot 当前已经形成较清晰的 daemon-first Agent Runtime 架构：

- `protocol` 作为共享协议与类型单一来源。
- `core` 承载 Agent Loop、状态机、审批、执行、响应、记忆等核心业务。
- `api` 承接 HTTP routes、JSON-RPC router、WebSocket 协议适配与连接 registry。
- `daemon` 聚焦进程启动、依赖装配、静态资源、恢复、metrics、WebSocket upgrade。
- `storage` 提供 in-memory 与 PostgreSQL 两套持久化实现。
- `skill-runner` 通过 `SkillToolExecutor` 统一接入 Agent Loop，不再有并行 workflow 执行器。
- `web` 作为 Chat UI 和 Agent timeline 消费端。

当前架构已摆脱旧 runtime 主路径和 workflow 抽象。旧 `packages/core/src/runtime/**`、daemon 旧 routes/ws 文件、`database.jobs` repository、`EventType` alias、硬编码 audit actor 等问题均已清理或收敛。`workflow_execution` intent 已替换为 `automation_execution`，`workflow.*` 伪 skill 路由已删除，`WorkflowToolExecutor`/`WorkflowRegistry`/`WorkflowRecord` 等类型全部移除。事件流已改为 raw event -> persist -> live event 的双 bus 模型。

综合判断：当前代码可以作为 Agent Runtime 架构优化后的可运行基线，整体健康度较高；后续重点是补齐 API 包测试、强化持久化/迁移测试、以及 Skill SDK 增强（respone body size limit、structured secret errors）。

| `packages/web` | React Chat UI、WS 客户端、timeline、approval、artifact、rich cards | 交互路径完整。 |
| `packages/launcher` | CLI/启动入口包装 | 职责单一。 |

### 3.2 主路径

当前标准交互路径：

```text
Web / CLI / HTTP
  -> packages/api HTTP routes 或 JSON-RPC router
  -> AgentService.handleChatCommand()
  -> AgentLoopEngine.run()
  -> Context -> Intent -> Plan -> Tool decision -> Safety -> Execution -> Reflection -> Response -> Memory
  -> SkillToolExecutor -> SkillRunner -> capability.handler()
  -> storage repositories
  -> raw eventBus -> RepositoryAgentEventSink -> liveEventBus -> WebSocket/UI
```

关键设计点：

- `AgentService` 是对话入口门面。
- `AgentLoopEngine` 是 Agent 状态机核心。
- `ExecutionOrchestrator` 直接注入 `SkillToolExecutor`，不再通过 bridge 分发。
- `RepositoryAgentEventSink` 持久化 raw event，补齐 DB sequence。
- Skill catalog 使用全限定 id 格式：`<skill-id>:<capability-name>`。

## 5. 主要优势

### 5.1 旧 runtime 主路径与 workflow 抽象清理完成

当前源码中未再发现：

- `SunPilotRuntime`
- `RepositoryRuntimeStore`
- `LegacyWorkflowMode`
- `runtimeStore`
- `workflowRuntime`
- `WorkflowToolExecutor`
- `WorkflowRegistry`
- `WorkflowRecord`
- `WorkflowPlan`
- `WorkflowStepPlan`
- `workflow_execution` intent
- `workflow.*` skillId 路由

### 5.2 Skill 是唯一工具抽象

- Skill catalog 使用全限定 id（`<skill-id>:<capability-name>`），解决同名 capability 冲突。
- `SkillToolExecutor` 直接注入 `ExecutionOrchestrator`，无中间 bridge。
- `ctx.http` 受控 HTTP API 已实现，支持 manifest network allow-list、audit、header redaction。

### 5.3 Agent event 语义收敛

事件类型已收敛到 `AgentEventType`，`EventType` alias 已删除。事件链路：

```text
rawEventBus.emit/publish
  -> async persist bridge
  -> database.events.append()
  -> liveEventBus.publish(persisted event)
  -> JSON-RPC / WebSocket notification
```

### 5.4 持久化层结构清晰

`DatabaseContext` 不再包含 `WorkflowRepository`。Postgres 与 in-memory 测试实现对齐。

## 6. 已解决的风险

### R4 (已解决). Workflow 概念已清理

原 "Workflow 当前还不是完整 workflow engine" 风险已通过删除 workflow 概念完全解决。Skill 现在是唯一的可执行 tool 抽象，自动化能力通过 Skill capability 表达。

### R6 (已解决). Core package 依赖面收窄

`@sunpilot/core` 不再依赖 `@sunpilot/workflow`，依赖面已收窄。

## 7. 当前风险与改进建议

### R1. `AgentEventBus.flush()` 解决竞态但抽象偏实现细节

（同前版）建议中期引入专用 `AgentEventDispatcher`。

### R2. `packages/api` 和 `packages/protocol` 缺少自身单测

（同前版）建议补充 schema、ws-protocol、agent-errors 测试。

### R3. migration 防漏机制仍是手工列表

（同前版）建议增加 migration list 防漏测试。

### R4. HTTP SDK 部分增强待完成

当前 `ctx.http` 已实现核心能力，但以下增强项待后续迭代：
- 响应体大小限制
- `AbortSignal.timeout(request.timeoutMs)` 注入
- runner.test.ts 中 HTTP allow/deny/redaction/arrayBuffer/signal 专项测试

优先级：P2。

## 8. 建议路线图

### 近期

- 给 `packages/api` 和 `packages/protocol` 补基础单测。
- 为 `AgentEventBus.flush()` 增加超时或错误策略。
- HTTP SDK 补充响应体大小限制和 timeout 注入。

### 中期

- 将 raw -> persist -> live 从 bus listener 模式提升为显式 dispatcher。
- 继续收敛 HTTP route body/query schema。

### 后续

- 若需要 composite capability（多步骤编排），在 SkillRunner 内部增加支持，不复活 workflow runtime。
- 若需要多进程/分布式 daemon，重新设计 event bus 与 live event delivery。

## 9. 最终评语

当前 SunPilot 架构已从 "daemon 内聚大量 runtime/route/ws 逻辑" 转向 "协议、核心、API、进程宿主、持久化、Skill 执行、前端 UI 分层协作" 的形态。Workflow 概念已完全清理，Skill 成为统一工具抽象。主路径清晰，121 个测试全部通过。剩余工作主要是工程韧性和 HTTP SDK 增强。整体评估为：架构方向正确，可作为稳定基线继续迭代。

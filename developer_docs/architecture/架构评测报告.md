# SunPilot 当前架构评测报告

更新日期：2026-06-28

评测口径：区分“文件存在”“在 composition root 装配”“实际经过运行路径”三种状态；结论以当前源码与测试为准。

## 1. 总体结论

SunPilot 当前架构方向正确，主链路已经是完整 Agent Runtime，不再是“单次模型调用 + UI”。`protocol / storage / platform / core / api / daemon / web` 分层明确，Agent Loop、工具、审批、事件、持久化、恢复、记忆和追踪已实际接入。

相比早期评测，以下能力已经落地，不能再列为缺失项：

- 附件从 Web 到 Agent input、消息 metadata、历史上下文与 retry/resume 的全链路传递；
- `ToolArgumentBuilder`、schema 校验/repair、native function calling 和结果投影；
- pgvector/hybrid memory、摘要压缩、query expansion、rerank、multi-hop 和 stale detection；
- context snapshot、trace/span、plan snapshot 与 model/tool call 观测；
- `permissionMode` 与 `modelId` 端到端传递；
- `agent.message.part.delta` 内容块流式协议；
- Digital World task 到 Agent Run、再回到 action/task completion 的桥接。

当前风险已从“主链路缺功能”转向“边界一致性与生产强化”：模型配置入口不统一、Skill 同进程执行、Digital World 编排仍固定、部分 API 合同和默认值存在漂移。

## 3. 架构强项

### 3.1 协议、业务与运行时边界清楚

共享命令和事件由 `protocol` 定义；REST/WS 只在 `api` 适配；普通产品业务在 `platform`；Agent 状态与决策在 `core`；对象创建、worker 和进程生命周期在 `daemon`。这种分层适合继续扩展桌面端或其他客户端。

### 3.2 Agent 事件先持久化再对外广播

内部组件写 raw event bus，持久化 sink 分配数据库 sequence，再进入 live event bus。高频 token delta 作为瞬态事件直接通过同步回调传输，避免异步落库导致乱序；最终 assistant message 保证内容可恢复。

### 3.3 工具链已经结构化

工具能力来自 Skill manifest/schema，候选经过检索和选择，模型可发 native tool call，参数会归一化、校验和修复，缺参可澄清。执行经 PermissionPolicy、Approval、ExecutionOrchestrator、SkillToolExecutor 和 SkillRunner，结果再做 projection 后进入后续上下文。

### 3.4 上下文与记忆不再是简单拼历史

当前上下文管线同时处理历史消息、附件、摘要、memory、压缩和 token budget；memory repository 支持向量/混合检索与关系，调用链带耗时和 snapshot。架构层面已具备长会话优化基础。

### 3.5 运行可观测性覆盖关键决策点

可通过 API 和 Run Debug Panel 查看 run、status history、events、tool calls、model calls、context snapshots、trace spans 和 plan snapshots。问题定位不再只能依赖 daemon stdout。

## 4. 当前问题清单

### P0：第三方 Skill 不是强隔离执行

`SkillRegistry.entryUrl()` 返回本地 ESM URL，`SkillRunner` 直接 `import()`。虽然 runner 提供受控 `ctx.files/http/secrets`、拒绝 manifest shell 权限并执行超时/并发控制，但恶意模块顶层代码仍与 daemon 同进程、同 Node 权限运行。

建议：

1. 未受信 Skill 迁移到 child process、Worker 权限隔离或容器沙箱；
2. 主进程只通过结构化 IPC 暴露 capability；
3. 安装时做签名、来源和 hash 校验；
4. 文档中不要把当前 runner 描述成安全的独立沙箱。

### P1：模型环境变量存在双入口不一致

daemon 创建 chat agent 时先执行 `createDefaultLlmProvider()`，该函数只读取 `SUNPILOT_LLM_API_KEY` / `DEEPSEEK_API_KEY` 和 shared base/model。composition root 随后又读取 `SUNPILOT_DP_LLM_*`。结果是 `SUNPILOT_DP_LLM_API_KEY` 单独配置仍启动失败。

建议统一为一个 `resolveModelConfig()`，由它同时服务 daemon bootstrap、ModelRouter、diagnostics 和 `/v1/models`。

### P1：模型列表默认值与可用性不一致

`/v1/models` 固定返回 `defaultModelId: "seed"`，而 Seed 可能 `available: false`。UI 应以“可用模型优先级”或服务端真实 route 结果作为默认值，不应固定不可用 id。

### P1：Digital World 仍是固定业务脚本

`TASK_ACTION_MAP` 只识别少量任务类型，按固定 action 序列推进；未知任务类型创建后没有通用 planner。Agent 只在 `work_on` action 被调用，尚不是由 Agent 自主生成并验证整个世界计划。

建议把“任务模板”和“Agent 计划”明确区分，新增可持久化 plan schema、action 前置条件、幂等 key 和人工确认点。

### P1：WS 与 REST Chat 能力不对称

WS `chat.send` 支持模型、权限、附件、异步 ack 和内容块流式；`POST /v1/chat` 只解析基础消息并同步等待结果。它适合作为简化 API，但当前命名容易让集成方误以为能力等价。

建议明确标注 REST 为 simple adapter，或扩展 request schema 并提供 SSE/异步 run 模式。

### P2：composition root 仍承担过多细节

Agent 大文件已经拆解，但 `createAgentLoopService()` 仍直接创建模型、embedding、context、tool、安全、memory、trace 及大量 adapter。dist 中出现过 composition 子模块，而 src 当前没有对应拆分，说明装配边界还可继续稳定。

建议按 `model/context/tool/safety/persistence` 拆纯工厂函数，每个工厂返回明确依赖集合并配装配测试。

### P2：路径模型保留已停用后端痕迹

`SunPilotPaths` 仍暴露 `db`、`analytics`、`vectors`，但运行数据库只允许 PostgreSQL，向量在 pgvector。兼容字段容易让维护者误判 SQLite/DuckDB/LanceDB 仍在运行。

建议标记 deprecated，并在下一次配置 schema 版本升级时移除。

### P2：评测资产不足

`evals/` 当前没有形成可复现的业务评测文件。工具选择已有 golden tests，但跨真实模型的回归、长会话 memory 召回、审批续跑、Digital World 任务闭环仍缺统一门禁。

建议建立固定数据集，报告 task success、tool/arg accuracy、clarification correctness、TTFT、总延迟和 token 成本。

## 5. 推荐路线

### 第一阶段：边界一致性

1. 统一模型配置解析与 `/v1/models` 默认值。
2. 明确 REST/WS Chat 合同与文档。
3. 清理路径模型中的停用后端语义。

### 第二阶段：安全生产化

1. 将 Skill 执行迁出 daemon 进程。
2. 为 Skill 包增加来源、hash、签名和版本锁定。
3. 对 token、Origin、OSS、approval 做部署级安全测试。

### 第三阶段：业务自主性与评测

1. 将 Digital World 固定模板升级为 plan/action contract。
2. 建立真实模型和业务工具 golden/eval suite。
3. 用评测数据调优 routing、memory、reflection 和 replanning，而不是继续堆启发式。

## 6. 最终判断

SunPilot 的长期架构目标没有偏：daemon 是本地可信控制面，Agent Core 是有状态执行内核，Skill 是扩展边界，Web 是可观测工作台。当前最重要的迁移步骤是把“应用层已经可靠”推进到“边界在生产环境也可信”：先统一模型配置，再隔离 Skill，最后用可重复评测提升 Digital World 和工具任务的自主完成率。

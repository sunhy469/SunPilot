# SunPilot 当前架构评测报告

更新日期：2026-06-20

评测对象：当前工作区实际代码，覆盖 `packages/protocol`、`packages/core`、`packages/api`、`packages/platform`、`packages/daemon`、`packages/storage`、`packages/skill-runner`、`packages/skill-sdk`、`packages/web`、`packages/launcher`、`tests/integration`，并重点复核 Agent Core、工具调用、工具参数、上下文构建、记忆、会话存储、审批、多轮动作和前端交互链路。

## 1. 总体结论

SunPilot 已经具备一个可运行的 daemon-first Agent Runtime 基线。当前主路径是：

```text
Web / REST / CLI
  -> api JSON-RPC / HTTP routes
  -> platform (产品业务) / core (Agent 能力)
  -> core AgentService
  -> AgentLoopEngine
  -> ContextBuilder / IntentRouter / Planner / ToolDecisionEngine
  -> PermissionPolicy / Approval
  -> ExecutionOrchestrator / SkillToolExecutor / SkillRunner
  -> Reflection / ResponseComposer / MemoryWriter
  -> storage repositories / PostgreSQL
  -> raw event bus -> persisted event bus -> WebSocket UI
```

整体架构方向正确：conversation CRUD 等产品后端业务已从 `packages/api` 下沉到 `packages/platform`，`api` 只保留传输协议层职责。旧 workflow 概念已经收敛为 Skill capability，Agent Loop、持久化、事件、审批、前端 timeline 形成闭环。最近的图片搜 1688 场景也验证了工具调用可以真实执行，并将 OSS 图片 URL 注入工具参数。

但当前系统还处在"能跑通主链路，但 Agent Core 仍偏规则化和启发式"的阶段。最大风险不在包边界，而在智能运行时质量：工具参数构造缺少 schema 驱动，反思层过浅，上下文没有摘要压缩和语义召回，会话附件在部分链路丢失，前端和后端仍有"请求完成才 ack"的同步倾向。这些问题会直接影响用户感知：用户以为 agent 在做事，但内部可能只是 LLM 总结、工具结果不完整、或者等待态出现不及时。

## 3. 当前架构画像

### 3.1 包职责

| 包                      | 当前职责                                                     | 评价                                                    |
| ----------------------- | ------------------------------------------------------------ | ------------------------------------------------------- |
| `packages/protocol`     | Agent 命令、事件、错误码、共享类型和 Zod schema              | 是协议边界，应继续保持为唯一协议源。                    |
| `packages/platform`     | 产品后端业务层：conversations 等 CRUD、tenant 上下文预留     | 新增层，承接 api 下沉的业务逻辑，可被多端复用。         |
| `packages/core`         | AgentService、AgentLoopEngine、LLM、工具、上下文、记忆、审批 | 核心集中，后续应把启发式逻辑升级为可测试策略。          |
| `packages/api`          | HTTP routes、JSON-RPC router、WS protocol、event streamer    | 职责清晰，仅保留传输适配，业务逻辑已下沉到 platform。    |
| `packages/daemon`       | composition root、Fastify、WS upgrade、metrics、recovery     | 装配集中，负责创建 platform services 并注入 apiDeps。   |
| `packages/storage`      | repository 接口、Postgres 实现、migration、测试 DB           | 存储面完整；memory/vector/summary schema 需要扩展。     |
| `packages/skill-runner` | Skill 扫描、执行、SDK context、受控 HTTP                     | 工具执行统一入口正确。                                  |
| `packages/skill-sdk`    | Skill 定义与测试 helper                                      | 应继续暴露 input schema、output schema 和 UI metadata。 |
| `packages/web`          | Chat UI、WebSocket、timeline、approval、artifact、附件上传   | 需要统一即时响应准则，避免等待网络请求才展示状态。      |
| `packages/launcher`     | 本地 CLI 启动和管理                                          | 边界清晰。                                              |

### 3.2 调用关系

```text
普通产品业务：
Client -> api -> platform -> storage

Agent 运行能力：
Client -> api -> core -> storage / model provider / skill runner

二者交汇：
api 或 platform 负责准备产品上下文；
core 只消费运行 Agent 所需的上下文，不依赖 platform 的普通 CRUD。
```

### 3.3 标准运行链路

```text
ChatComposer / WelcomeView
  -> useChat.send()
  -> optimistic user message + pending state
  -> WebSocket chat.send
  -> JsonRpcRouter.handle()
  -> AgentService.handleChatCommand()
  -> create conversation / message / run
  -> publish agent.run.created
  -> AgentLoopEngine.run()
  -> ContextBuilder.build()
  -> IntentRouter.route()
  -> RuleBasedPlanner.createPlan() when needed
  -> ToolDecisionEngine.decide()
  -> PermissionPolicy.evaluate()
  -> ExecutionOrchestrator.execute()
  -> SkillToolExecutor.execute()
  -> SkillRunner.execute()
  -> BasicReflectionEngine.reflect()
  -> ResponseComposer.streamAndSave()
  -> MemoryWriter.writeFromTurn()
  -> agent.run.completed
```

## 4. 已确认的能力

### 4.1 工具真实执行和日志可追溯

以 2026-06-15 的 1688 图片搜索为例：

- run：`run_3c110651-c24e-4dab-95a2-c03e5aabec14`
- tool call：`tc_439aa5ab-e3dd-4dc6-8f21-b875995c4273`
- skill：`jaderoad:product.source.search1688`
- arguments 包含 `query`、`imageUrl`、`image_url`、`attachments`
- result 包含 `totalResults: 698` 和候选商品列表
- response 由 `deepseek-v4-flash` 基于工具结果总结生成

结论：当前工具结果可通过 `tool_calls`、`steps`、`events`、`model_calls` 反查，不是黑箱。

### 4.2 Agent Loop 已不是单次 LLM 调用

`AgentLoopEngine` 当前包含：

- context building
- intent routing
- optional planning
- tool deciding
- permission / approval
- tool execution
- reflection
- response
- memory writing
- error / cancellation handling
- approval resume
- reflection 返回 `continue` 时最多 5 轮继续工具决策

这已经满足"用户目标由 agent loop 控制，不是 LLM 文本结束就结束"的基本要求。

### 4.3 事件流具备持久化和实时推送

当前设计是：

```text
rawEventBus.emit()
  -> RepositoryAgentEventSink.persist()
  -> liveEventBus.publish()
  -> WebSocket event streamer
```

`agent.response.delta` 是 transient，不持久化，最终 assistant message 持久化到 `messages`。

### 4.4 Platform 业务层已就位

conversation 的列表、详情、历史消息、更新、touch、删除等产品 CRUD 已从 `packages/api/src/http/register-routes.ts` 下沉到 `packages/platform/src/conversations/conversation.service.ts`。API 层只保留 schema parse、context 解析和错误映射。`PlatformRequestContext` 已预留 `tenantId`、`userId`、`actorType`、`clientType` 字段，为多租户和多端复用做好准备。

## 5. 关键问题清单

### P0-1. `chat.send` 后端 ack 等待完整 Agent Loop 完成

位置：`packages/api/src/ws/json-rpc-router.ts`

当前 `JsonRpcRouter.handle("chat.send")` 会：

```text
await agent.handleChatCommand(...)
return { accepted: true, ... }
```

如果工具和模型耗时较长，JSON-RPC result 会在 run 完成后才返回。虽然前端当前已经 optimistic 显示 user message 和 pending，但协议语义上仍不健康：`chat.send` 应只表示"请求已接收"，不应等待任务完成。

建议：

- `AgentService` 增加 `startChatCommand()` 或 `enqueueChatCommand()`。
- `chat.send` 在创建 conversation/message/run 并发布 `agent.run.created` 后立即 ack。
- Agent Loop 在后台执行，所有状态通过 `agent.*` event 更新。
- HTTP `/v1/chat` 如果保留同步模式，应改名或单独提供 `/v1/chat/stream`。

### P0-2. 装配层丢失用户消息附件

位置：`packages/daemon/src/composition-root.ts`

`AgentService.handleChatCommand()` 调用 `conversations.createMessage({ attachments })`，但 composition root 的 `createMessage` 适配器没有把 `attachments` 传给 `database.messages.create()`。

影响：

- 新消息虽然前端本地显示附件，但数据库 message metadata 可能没有附件。
- 历史会话 replay 可以丢失附件。
- 后续 ContextBuilder 无法从历史消息恢复附件。

建议：

- 修正 composition root 的 `conversations.createMessage`，传入 `attachments: input.attachments`。
- 为 `repository-conversation.service`、composition root 增加附件持久化测试。

### P0-3. ContextBuilder 历史消息不包含附件 metadata

位置：`packages/core/src/agent-kernel/context/context-builder.ts`、`packages/storage/src/repositories/message.repository.ts`

`MessageRepository` 存了 metadata，但 `ContextBuilderDeps.listMessages` 只返回：

```ts
(id, role, content, createdAt);
```

影响：

- 当前消息附件能进入上下文，历史附件不能。
- 多轮对话中用户说"用刚才那张图继续搜"时，agent 看不到历史图片 URL。

建议：

- `listMessages` 返回 attachments。
- `ContextMessage.metadata.attachments` 保留附件数组。
- `ResponseComposer.buildMessages()` 对历史用户消息也注入附件摘要。
- 附件应有专门的 `attachment_index` 或 artifact 引用。

### P0-4. 工具参数构造仍是启发式

位置：`packages/core/src/agent-kernel/tools/tool-decision-engine.ts`

当前 `buildToolArguments()` 通过文本和附件启发式生成：

- `query`
- `imageUrl`
- `image_url`
- `attachments`
- `url` / `urls`

这解决了 1688 图片搜索，但不是通用方案。不同 skill 可能需要 `imageDataUrl`、`offerId`、`prompt`、`style`、`script`、`duration` 等参数，当前没有根据 capability input schema 生成。

建议：

- Skill manifest 暴露 capability input schema。
- ToolDecisionEngine 输出 `arguments` 前调用 `ToolArgumentBuilder`。
- `ToolArgumentBuilder` 使用 schema + 当前上下文 + 附件 + LLM structured output 生成参数。
- 执行前用 schema 校验；失败时进入参数修复回合，不直接执行或编造。

### P0-5. Reflection 太弱，无法判断业务目标是否完成

位置：`packages/core/src/agent-kernel/reflection/basic-reflection-engine.ts`

当前逻辑：

- 有 failed tool call -> `goalAchieved=false`，但 `nextAction=respond`
- 全部 completed -> `goalAchieved=true`，`nextAction=respond`

影响：

- 工具返回"需要继续查详情"也会被视为完成。
- `MAX_TOOL_ITERATIONS` 的继续能力很少被真实触发。
- Agent 多轮动作缺少目标状态机。

建议：

- 引入 `ReflectionDecision`：`goalAchieved`、`confidence`、`missingInfo`、`nextToolCandidates`、`shouldAskUser`、`stopReason`。
- 对结构化 tool result 做规则判断，例如搜索结果需要 detail、detail 需要 SKU/运费等。
- 必要时使用轻量 LLM 做"是否完成用户目标"的结构化判断。

### P1-1. ResponseComposer 容易对工具结果过度改写

位置：`packages/core/src/agent-kernel/response/response-composer.ts`

真实日志显示：工具返回第 2 条是男款衬衫，但 assistant 回复跳过它，选择第 3、4 条，并总结"都是女装"。这说明 LLM 总结可能选择性改写。

建议：

- 对工具结果类回复使用结构化 renderer 或模板优先。
- LLM 只负责解释和建议，不负责重排或补数据。
- Prompt 明确要求"只能引用工具返回字段，不得新增、改价、改销量、改链接"。
- 保存 response provenance：每个展示项对应 `toolCallId` 和 `candidate.id`。

### P1-2. Tool result 被 stringify 后整段塞入上下文

位置：`SkillToolExecutor.summarizeOutput()`、`ExecutionOrchestrator`、`ResponseComposer.composeFromObservation()`

大 JSON 结果会作为字符串进入 prompt，风险：

- token 浪费。
- LLM 解析不稳定。
- 后续工具决策难以使用结构化字段。

建议：

- `NormalizedToolResult` 持久化 `structured` 字段。
- ToolCallSummary 增加 `structuredPreview` 或 `resultRef`。
- ResponseComposer 用 top N + schema projection，不塞全量 JSON。

### P1-3. 记忆检索仍是 ILIKE，缺语义召回

位置：`PostgresMemoryRepository.search()`

问题：

- 中文同义词、跨语言、语义相关无法召回。
- 搜索 query 只来自当前消息，缺意图扩展。

建议：

- 引入 pgvector。
- memory、conversation summary、artifact summary 建 embedding。
- ContextBuilder 混合召回：keyword + semantic + recency + importance。

### P1-4. 缺长会话 compaction

位置：ContextBuilder / MemoryWriter / storage schema

当前 history 只取 30 条，超限丢弃，没有摘要层。

建议：

- conversation summary memory。
- rolling summary：每 N 轮或 token 超限触发。
- summary 保留事实、偏好、工具结果、决策、未完成事项、附件引用。

### P1-5. 前端上传和发送体验还不够"即时"

位置：`useFileAttachments`、`useOssUpload`、`ChatComposer`、`useChat`

已有优点：文件一加入就显示 uploading entry，消息发送后本地追加 user message。

待优化：

- 发送时不应因为 OSS 上传慢而让用户感觉"没反应"。
- 可以允许附件先作为 local pending attachment 发送 UI 状态，后台上传完成后补 URL，再正式启动后端 run。
- AI 等待态应在点击发送的同步栈内立即出现，不依赖 WebSocket open、ack、run.created。

准则：所有请求背后默默完成，前端先响应，后端自动推进。

### P1-6. 审批拒绝后 run 保持 waiting_approval

位置：`AgentService.reject()`

当前注释明确：拒绝后 run 保持 `waiting_approval`，允许用户手动取消。

建议：

- 拒绝后可根据策略转为 `interrupted` 或 `cancelled`。
- UI 提供"修改参数后重试""取消 run""继续无此工具"的明确路径。
- 审批事件应包含被审批工具的参数摘要和风险原因。

### P2-1. 工具风险策略偏粗

位置：`PermissionPolicy`

当前基于 capability category 和参数粗判。建议：

- 每个 capability manifest 提供 `riskHints.destructiveArgs`、`externalHosts`、`writesTo`。
- 权限策略支持用户设置：ask/auto/full。
- 审批策略考虑 host、文件路径、金额、发送对象、数据外流。

### P2-2. 可观测性缺 context snapshot

当前已有 events/tool_calls/model_calls，但缺：

- 每次模型调用的 context chunk 列表。
- 被裁剪 chunk 列表和原因。
- 工具参数生成过程。
- response 引用哪些 tool result。

建议新增 `context_snapshots` 或写入 `model_calls.metadata.context`.

## 6. 优化路线图

### 第一阶段：修正体验与数据丢失 (大部分已完成)

1. ✅ Platform 业务层抽取：conversation CRUD 从 api 下沉到 platform。
2. `chat.send` 改为快速 ack + 后台 run。
3. 修复 composition root 附件持久化。
4. ContextBuilder 历史消息支持附件。
5. 前端统一 optimistic pending：发送、上传、等待 AI 均立即显示。
6. ResponseComposer 对工具结果使用"只引用工具字段"的约束。

### 第二阶段：工具参数和多轮动作升级

1. 引入 `ToolArgumentBuilder`。
2. capability input schema 驱动参数生成和校验。
3. 参数缺失时自动澄清或修复。
4. ReflectionEngine 结构化判断目标是否完成。
5. Tool result 保存 structured，后续工具可直接消费。

### 第三阶段：上下文和记忆系统升级

1. pgvector + hybrid retrieval。
2. conversation rolling summary。
3. context snapshot 可观测性。
4. prompt cache friendly chunk ordering。
5. 多媒体附件索引和 OCR/图片描述缓存。

### 第四阶段：策略与治理

1. 更细粒度审批策略。
2. 用户权限模式真实接入后端。
3. 多租户数据隔离（基于 PlatformRequestContext.tenantId）。
4. 数据外流审计。
5. 长任务后台队列和恢复。

## 7. 最终评语

SunPilot 当前架构已经从"能调用模型的聊天应用"进化为"有状态、有工具、有审批、有持久化、有事件流的 Agent Runtime"。平台业务层（platform）的引入使 api 更薄、职责更清晰，为后续 Mac/Windows/Mobile 多端复用和多租户扩展打下基础。下一阶段的核心不是再堆功能，而是把 Agent Core 的关键决策点从启发式升级为结构化、可验证、可观测的机制：工具参数要 schema-aware，上下文要可检索可压缩，反思要能决定是否继续动作，前端要做到用户操作立即响应、后端任务后台推进。完成这些之后，SunPilot 才会真正从"会答"变成"可靠地办事"。

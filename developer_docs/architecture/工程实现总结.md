# SunPilot 当前工程实现总结

更新日期：2026-06-28

版本基线：`0.1.0`，以当前 `main` 工作区代码为准。

## 1. 项目定位与成熟度

SunPilot 是 daemon-first、local-first 的业务 Agent 运行时。它不是单纯的聊天 UI：用户消息进入持久化 Agent Run，经上下文、意图、规划、工具、审批、执行、反思、回复和记忆链路后，以事件流回传到 Web。

当前成熟度可分为三层：

| 层级                              | 状态                 | 说明                                                            |
| --------------------------------- | -------------------- | --------------------------------------------------------------- |
| 主对话与 Agent Runtime            | 已形成可运行闭环     | WS 对话、流式内容块、工具、审批、持久化、恢复、追踪均已接线     |
| 记忆、模型与安全治理              | 已实现，仍需生产验证 | pgvector、摘要压缩、双模型、注入检测、策略和 trace 均在运行路径 |
| Digital World 与第三方 Skill 隔离 | 实验/MVP             | 业务数据已持久化，但任务模板固定；Skill 仍与 daemon 同进程      |

## 2. Monorepo 边界

| 包                      | 实际职责                                                           |
| ----------------------- | ------------------------------------------------------------------ |
| `packages/protocol`     | Agent 命令、事件、错误、Zod schema、message part、rich card        |
| `packages/storage`      | PostgreSQL/pgvector、22 个迁移、repository 接口与实现、运行目录    |
| `packages/platform`     | Conversation 与 Digital World 产品业务服务                         |
| `packages/core`         | AgentService、AgentLoopEngine、LLM、上下文、工具、记忆、安全、追踪 |
| `packages/api`          | REST、WebSocket JSON-RPC、OSS 适配器；不负责核心业务决策           |
| `packages/daemon`       | 唯一装配根、Fastify 生命周期、静态 Web、恢复、后台 worker、metrics |
| `packages/web`          | React Chat、Settings、Run Debug、附件、Digital World               |
| `packages/skill-sdk`    | Skill 定义与测试 helper                                            |
| `packages/skill-runner` | Skill 扫描、manifest 校验、动态加载、受控执行                      |
| `packages/launcher`     | `sun` / `sunpilot` CLI                                             |

依赖方向的目标是：协议和存储在底层，产品业务在 `platform`，Agent 行为在 `core`，传输在 `api`，对象创建只在 `daemon`。

## 3. 两条主运行链路

### 3.1 对话/Agent 链路

```text
ChatComposer
  -> useChat
  -> WebSocket chat.send
  -> api JsonRpcRouter
  -> AgentService.startChatCommand
  -> AgentLoopEngine.run
  -> rawEventBus
  -> RepositoryAgentEventSink
  -> liveEventBus
  -> WebSocket event streamer
  -> useChat 合并本地消息、持久化消息和流式 part
```

`chat.send` 的完整参数包括 `clientRequestId`、`conversationId`、`message`、`mode`、`permissionMode`、`modelId` 和 `attachments`。附件会随用户消息和 run input 持久化，retry/resume 可恢复原始附件。

流式回复采用内容块协议：

- `agent.message.started`
- `agent.message.part.started`
- `agent.message.part.delta`
- `agent.message.part.updated`
- `agent.message.completed`

高频 `agent.message.part.delta` 不写数据库，最终消息写库；其他运行事件先持久化获得 sequence，再广播给外部消费者。

### 3.2 普通产品业务链路

```text
Web / client -> api route -> platform service -> storage repository
```

Conversation CRUD 与 Digital World 服务不放进 Agent Core。`platform` 还通过窄接口调用 AgentService，使数字生命任务可以创建异步 Agent Run，而不反向依赖完整 core 类型。

## 4. Agent Core 当前实现

### 4.1 入口与状态

`AgentService` 负责校验请求、幂等、会话与消息持久化、run 创建、异步启动、停止/取消/恢复/重试和审批动作。重实现已拆到 `agent-service/agent-service-implementation.ts`，公开文件保留稳定导出。

`AgentLoopEngine` 管理 run 生命周期与主循环，复杂分支已拆为：

- `run-preparation.ts`：run 初始化、上下文与前置准备；
- `preliminary-inference.ts`：前置推理；
- `approval-flow.ts` / `approval-continuation.ts`：审批暂停和续跑；
- `run-outcomes.ts`：完成、失败、取消等终态；
- `utils.ts`：共享辅助逻辑。

运行状态、事件、status history、plan snapshot、model/tool call 和 trace/span 均有持久化接口。

### 4.2 上下文与记忆

`ContextBuilder` 当前包含：

- 当前消息和附件 metadata；
- 会话历史与 rolling summary；
- stale summary 检测；
- memory hybrid/vector 搜索；
- query expansion、multi-hop retrieval、MMR rerank；
- source compression 和 token budget；
- 每次模型调用的 context snapshot metadata。

Embedding 有真实 provider 时使用远端向量；否则回退到 keyword/hash 向量。记忆写入包含策略判断、脱敏、去重/关系、重试和质量字段。

### 4.3 意图、规划与工具

意图路由采用分层级联：表单/规则匹配、embedding 高置信短路、LLM 分类。Skill embedding cache 在装配层共享，reload 时会失效。

工具链当前同时支持：

- 候选 skill 检索与 schema 定义；
- OpenAI-compatible native function calling；
- 受控文本函数调用解析兜底；
- schema 参数归一化、补全、校验和 repair；
- 图片附件从当前消息或历史上下文补入；
- structured tool result 投影，避免整段结果无限进入上下文；
- 多轮工具调用、replanning、reflection 与 clarification。

规划仍以 `RuleBasedPlanner` 为基线，配合 `PlanValidator` 和 `Replanner`。这意味着结构已经完整，但规划智能度仍受规则和模型质量影响。

### 4.4 审批与安全

当前运行路径接入：

- `PermissionPolicy` 与 `ask / auto / full`；
- repository-backed approval gate、request、decision、expiry；
- `PromptInjectionDetector`；
- `ToolSandbox` 的 strict/moderate/permissive 策略；
- `TaskScopedPermissionManager`；
- 本地 bearer token、Origin 限制和滑动窗口限流。

需要明确：Skill 使用 ESM dynamic import 进入 daemon 进程。Runner 会拒绝 shell，约束文件路径、HTTP host 和 env allowlist，但它还不是进程级或容器级安全沙箱。

## 5. 模型路由

`ModelRouter` 注册两个同级业务模型：

- `dp`：默认 DeepSeek/OpenAI-compatible provider；
- `seed`：配置 `SUNPILOT_SEED_LLM_API_KEY` 后启用的 Volcengine Ark provider。

Web 选择器的 `modelId` 已经贯穿 `ChatComposer -> useChat -> chat.send -> AgentService -> AgentLoopInput -> ModelRouter`。Model call 会记录 purpose、provider、model、token、latency、first token 和 context metadata。

当前有一项配置兼容性债务：daemon 先调用 `createDefaultLlmProvider()`，它只认 `SUNPILOT_LLM_API_KEY` 或 `DEEPSEEK_API_KEY`。因此只设置 `SUNPILOT_DP_LLM_API_KEY` 仍无法启动；文档与部署必须以 shared key 为必需项，直到 bootstrap 统一。

## 6. 存储与后台任务

运行数据库仅支持 PostgreSQL，Docker 镜像为 `pgvector/pgvector:pg16`。迁移覆盖：

- conversations / messages；
- runs / steps / events / status history / idempotency；
- tool calls / model calls / approvals / artifacts / audit；
- memory、embedding、summary stale 和关系；
- trace / span / plan snapshot；
- Digital World nodes、edges、beings、tasks、actions、logs、artifacts。

daemon 启动时执行迁移、审批过期处理和异常 run 恢复，并启动 stale detection 与 memory pruning worker。

`~/.sunpilot/sunpilot.db`、`analytics`、`vectors/lance` 是早期路径兼容字段，不代表当前存在 SQLite、DuckDB 或 LanceDB 运行后端。

## 7. Web 产品面

主要页面是 Chat 和 Settings。Chat 已包含：

- optimistic user message、pending 状态与断线重连；
- 流式 Markdown/rich card；
- conversation 列表、置顶、重命名和删除；
- 附件上传进度、失败重试、OSS URL 门禁；
- model 与 permission mode 选择；
- approval、artifact、tool timeline、Run Debug Panel。

Digital World 使用 PixiJS，已支持平移缩放、节点和多数字生命、状态气泡、路径动画、面板、快捷键、主题、音效和 dev editor。生产数据来自 `/v1/digital-world`；开发环境在接口不可用或空数据时允许 mock fallback。

## 8. 当前真实边界

1. `REST /v1/chat` 是精简同步适配器，没有完整透传附件、模型与权限模式；产品完整能力以 WebSocket 为准。
2. Digital World 的任务种类由 `TASK_ACTION_MAP` 固定，未知类型不会形成通用自主规划。
3. Skill 无 OS 级隔离，第三方不可信代码不能仅依赖当前 allowlist 视为安全。
4. 双模型 bootstrap 仍有 shared/DP 环境变量不一致。
5. `/v1/models` 当前固定返回 `defaultModelId: seed`，即使 Seed 未配置也会暴露默认值语义不一致。
6. `composition-root.ts` 仍较大；核心巨型文件已拆分，但装配模块仍可继续按 context/model/tool/safety/persistence 分组。

## 10. 结论

SunPilot 已经跨过“聊天壳”阶段，主架构是有状态、可恢复、可审批、可观测的 Agent Runtime。近期最值得投入的不是再补同类模块，而是收紧四个边界：统一模型配置、隔离不可信 Skill、把 Digital World 从固定任务模板提升为可验证的任务编排、用真实场景评测持续校准工具与记忆质量。

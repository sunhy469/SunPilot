# SunPilot 当前工程实现总结

更新日期：2026-06-20

本文档总结当前 SunPilot 工程实现、核心模块边界、Agent 主链路、前端交互链路和后续改造入口。

## 1. 项目定位

SunPilot 是一个 daemon-first 的本地业务 Agent 运行时，目标是让用户通过 Web Chat 发起任务，由后台 Agent Loop 自动完成上下文构建、工具选择、工具执行、审批、反思、多轮动作、回复生成和记忆沉淀。

当前能力覆盖：

- Web Chat 对话入口。
- 文件/图片附件上传到 OSS 并随消息进入 Agent 输入。
- Agent Loop：context、intent、planning、tool decision、permission、execution、reflection、response、memory。
- Skill 生态：安装、扫描、启用、capability 执行、受控 HTTP。
- PostgreSQL 持久化：conversation、message、run、event、approval、artifact、memory、tool call、model call、audit、status history。
- WebSocket 事件流：实时 delta、tool timeline、approval、artifact、run 状态。
- CLI launcher：启动 daemon、打开 Web、状态检查。

## 3. 核心运行架构

```text
Browser
  |
  | HTTP / WebSocket
  v
packages/api
  - REST routes (薄适配器)
  - JSON-RPC router
  - event streamer
  |
  +--> packages/platform          +--> packages/core
  |      - ConversationService    |      - AgentService
  |      - DTO mapping            |      - AgentLoopEngine
  |      - 分页/业务逻辑            |      - ContextBuilder
  |      |                        |      - ToolDecisionEngine
  |      v                        |      - ExecutionOrchestrator
  |    packages/storage           |      - ResponseComposer
  |      - PostgreSQL             |      - MemoryWriter
  |      - conversations/messages |      |
  |                               |      v
  |                               |    packages/storage
  |                               |      - runs/events/tool_calls
  |                               |      - memory/approvals/artifacts
  |                               |
  |                               +--> packages/skill-runner
  |                                      - Skill registry
  |                                      - Skill execution
  |                                      - SDK context
```

### 3.1 产品业务路径

```text
Client -> api -> platform -> storage
```

conversation 的列表、详情、历史消息、更新、删除等操作通过 `platform.conversations` 完成。API 层只做 schema parse、context 解析和错误映射。

### 3.2 Agent 运行路径

```text
Client -> api -> core -> storage / skill-runner
```

chat.send、runs、approvals、tools、events 等 Agent 运行时路径继续由 `api -> core` 处理，不经过 platform。

## 4. Agent Core 主链路

### 4.1 AgentService

文件：`packages/core/src/agent/agent.service.ts`

职责：

- 校验 conversation。
- 创建 conversation / user message / run。
- 处理幂等性。
- 创建 AbortSignal。
- 订阅 raw event bus 和 live event bus。
- 将 transient delta 推给当前连接。
- 调用 `AgentLoopEngine.run()`。
- 处理 stop/cancel/resume/retry/approve/reject。

注意：当前 `handleChatCommand()` 会等待完整 Agent Loop 完成后才 resolve。WebSocket `chat.send` 虽然有事件流，但 JSON-RPC ack 不是快速 ack。后续应把 run 创建和后台执行拆开。

### 4.2 AgentLoopEngine

文件：`packages/core/src/agent-kernel/agent-loop-engine.ts`

当前状态机：

```text
context_building
  -> intent_routing
  -> planning?
  -> tool_deciding
  -> no_tool | ask_clarification | require_approval | use_tool
  -> executing
  -> reflecting
  -> continue? tool_deciding : responding
  -> memory write
  -> completed
```

已具备：

- no_tool 直接生成回复。
- use_tool 执行工具后再回复。
- ask_clarification 直接向用户提问。
- require_approval / waiting_approval 暂停。
- approval approve 后 `resumeApprovedTool()` 继续执行。
- 工具后 reflection 返回 `continue` 时最多 5 轮继续执行。
- error / cancellation 统一转成 AgentLoopResult。

主要缺口：

- 继续执行完全依赖 `BasicReflectionEngine`，而当前 reflection 很弱。
- `handleApprovalRequired()` 的 requestedAction 是空占位，不适合由工具决策阶段直接产生审批。
- rejection 后 run 状态仍停留在 waiting_approval。

### 4.3 ContextBuilder

文件：`packages/core/src/agent-kernel/context/context-builder.ts`

当前上下文来源：

| 来源                 | priority | 当前行为                    |
| -------------------- | -------- | --------------------------- |
| system persona       | 0        | 必留                        |
| system rules         | 0        | 必留                        |
| safety policy        | 0        | 必留                        |
| current message      | 0        | 必留                        |
| conversation history | 10       | 取最近 30 条                |
| memories             | 15       | ILIKE 检索，最多 10 条      |
| tool results         | 18       | 读取当前 run 的 tool calls  |
| skill catalog        | 20       | 列出已启用 skill capability |
| artifacts            | 25       | 当前 run artifacts          |
| run state            | 0        | 必留                        |

关键问题：

- history 只有 content，没有附件 metadata。
- 消息数量固定 30，不按 token 或重要性动态裁剪。
- 没有 conversation summary。
- 没有 semantic retrieval。
- tool result 大 JSON 可能直接进入 prompt。
- token 估算是字符数 / 4。

### 4.4 IntentRouter

文件：`packages/core/src/agent-kernel/intent/intent-router.ts`

当前策略：

- 先规则匹配。
- 规则未命中时用轻量 LLM 分类。
- use_skill 可返回 selected skill id。

优点：常见 intent 快速、可控。

缺口：

- 规则容易误召回。
- use_skill 依赖 skill catalog 文本。
- 缺多意图拆解，例如"搜同款并生成营销脚本"。

### 4.5 ToolDecisionEngine

文件：`packages/core/src/agent-kernel/tools/tool-decision-engine.ts`

当前策略：

1. plan 中有 tool step 就直接执行。
2. intent 不需要工具则 no_tool。
3. candidateSkills 匹配 availableSkills。
4. use_skill 且无 candidate 时做确定性评分。
5. 歧义时可 ask_clarification。
6. LLM 可从 top candidates 中选 skill。
7. fallback 到 `INTENT_SKILL_MAP`。

当前工具参数：

- plan step 有 input 时直接使用。
- 否则调用 `buildToolArguments()`。
- `buildToolArguments()` 会从当前消息和附件生成 `query`、`attachments`、`imageUrl`、`image_url`、`url`、`urls`。

缺口：

- 没有读取 capability input schema。
- 没有参数校验和自动修复。
- 没有参数生成 provenance。
- 没有针对多工具链路的参数继承，如 search -> detail。

### 4.6 ExecutionOrchestrator 与 SkillToolExecutor

文件：

- `packages/core/src/agent-kernel/execution/execution-orchestrator.ts`
- `packages/core/src/agent-kernel/execution/skill-tool-executor.ts`

当前行为：

- 按 skill category 做并发控制。
- 支持 retry。
- 创建 `tool_calls` 记录。
- 委托 `SkillToolExecutor`。
- `SkillToolExecutor` 解析 `<skill-id>:<capability-name>`，创建 step，执行 SkillRunner，收集 artifacts。

缺口：

- `summarizeOutput()` 对对象直接 JSON.stringify。
- `NormalizedToolResult.structured` 没进入 `ToolCallSummary`。
- step capability 在 composition root 创建 step 时未传完整 capability 字段。

### 4.7 Reflection

文件：`packages/core/src/agent-kernel/reflection/basic-reflection-engine.ts`

当前逻辑：

- 任意 tool failed：goalAchieved=false，但 nextAction=respond。
- 全部 completed：goalAchieved=true，nextAction=respond。

这是当前 Agent 多轮动作质量的核心短板。它只能判断工具是否完成，不能判断用户目标是否完成。

### 4.8 ResponseComposer

文件：`packages/core/src/agent-kernel/response/response-composer.ts`

当前行为：

- `composeDirect()` 用 AgentContext 生成普通回复。
- `composeFromObservation()` 将工具 summary 和 reflection 拼成 system message，让 LLM 总结。
- `agent.response.delta` 实时发出。
- 完成后保存 assistant message。

风险：

- 工具结果被 LLM 改写、跳过或过度总结。
- 没有强制引用 tool result 字段。
- 没有 response provenance。

### 4.9 MemoryWriter

文件：`packages/core/src/agent-kernel/memory/memory-writer.ts`

当前候选来源：

- 显式"记住"。
- `memory_update` intent。
- 工具任务完成后写 conversation summary。

当前策略：

- secret scan。
- 搜索相似记忆。
- MemoryPolicy 决定 write/supersede/reject。

缺口：

- summary 粒度粗。
- 没有 embedding。
- 没有重要事件和长期目标的主动抽取。
- 没有按会话阶段压缩历史。

## 5. Platform 业务层

### 5.1 设计

文件：
- `packages/platform/src/conversations/conversation.service.ts`
- `packages/platform/src/conversations/conversation.types.ts`
- `packages/platform/src/conversations/conversation.mapper.ts`
- `packages/platform/src/context.ts`

### 5.2 集成

daemon 启动时：

```ts
const platform = createPlatformServices({ database });
const apiDeps = { database, platform, ... };
registerSunPilotApiRoutes(app, apiDeps);
```

api 层会话路由改为薄适配器：

```ts
// 之前：直接调用 database.conversations + 手动分页/DTO映射
// 现在：schema parse -> platform.conversations.listConversations(context, input)
```

## 6. 存储实现

### 6.1 messages

文件：

- `packages/storage/src/repositories/message.repository.ts`
- `packages/storage/src/postgres/postgres.message.repository.ts`

messages 表将 attachments 存在 metadata 中。HTTP `/v1/conversations/:id/messages` 会把 metadata.attachments 映射给前端（现在由 `conversation.mapper.ts` 处理）。

当前问题：

- composition root 的 `conversations.createMessage` 没传 attachments。
- ContextBuilder 的 listMessages 适配器没返回 attachments。

### 6.2 runs / events / tool_calls / model_calls

当前已能支持：

- run 状态查询。
- event replay。
- tool call 参数和结果审计。
- model call token 和 latency。
- status history。

建议补充：

- context snapshot。
- prompt snapshot。
- tool argument generation trace。
- response provenance。

### 6.3 memory

当前 `memory_metadata` 支持 scope、type、confidence、importance、metadata、expires/supersede/delete。

短板：

- 搜索仍是 ILIKE。
- 无 pgvector。
- 无 conversation summary 专表或 summary chunk 类型。

### 6.4 migrations

当前有 19 个 SQL migration 文件 (001—019)，覆盖 init、conversations、messages、runtime aux/steps、catalog、agent runtime core、events sequence、idempotency、observability、memory core、agent runtime cleanup、conversations kind、pgvector embeddings、model calls metadata、tool call metadata、agent trace plan、memory stale columns、conversations pinned。

## 7. 前端对话实现

### 7.1 当前链路

```text
ChatComposer
  -> useFileAttachments
  -> useOssUpload
  -> useChat.send()
  -> optimistic user message
  -> pending + thinking
  -> WebSocket chat.send
  -> agent.response.started / delta / completed
```

已有优点：

- 文件加入后立即显示 UploadFile entry。
- 发送后立即追加本地 user message。
- pending 状态在 send 同步执行时设置。
- response.started 时创建 assistant 空消息。
- delta 到达时追加内容。

### 7.2 必须坚持的交互准则

所有用户请求都应表现为"前端即时响应，后端后台推进"：

- 用户选择文件：立即显示附件卡片，不等 OSS。
- 用户拖拽/粘贴图片：立即显示本地预览和上传进度。
- 用户点击发送：立即显示用户消息和 AI 等待态，不等 WebSocket open，不等 JSON-RPC ack，不等 run.created。
- OSS 上传慢：前端显示"附件上传中/等待后端处理"，后台继续上传并在 URL 可用后启动或补全请求。
- Agent 执行工具：timeline 更新，不阻塞等待点动画。
- 模型尚未输出 delta：AI 消息占位和 typing dots 必须已经可见。

### 7.3 当前需要优化的点

- `ChatComposer` 在 `uploading` 时禁止发送，体验上会让用户觉得请求没发出去。可以保留数据安全，但需要即时反馈"上传完成后自动发送"。
- `chat.send` 后端 ack 太晚，协议上应该改成快速 accepted。
- pending 视觉不应依赖 first delta。
- 附件上传错误应保留本地卡片并提示可重试，不应静默丢失。

## 9. 建议的近期实现顺序

1. ✅ Platform 业务层抽取：conversation CRUD 从 api 下沉到 platform。
2. 修复附件持久化和历史附件上下文。
3. 将 `chat.send` 改成快速 ack + 后台 run。
4. 前端增加"上传完成后自动发送"和更明确的 AI 等待占位。

6. 将 tool result structured 化，ResponseComposer 只投影必要字段。

8. 引入 conversation summary 和 pgvector 语义记忆。

## 10. 当前工程判断

当前工程的骨架是对的，状态机、事件、存储、工具执行、前端消费都已经连通。Platform 业务层的引入使 api 更薄、职责更清晰，产品业务逻辑下沉到 platform 后可被 Mac/Windows/Mobile 多端复用。接下来最重要的是提升"可靠办事"的质量：用户发出目标以后，前端要立刻响应，后端要后台持续推进；Agent Core 要根据结构化上下文、结构化参数、结构化工具结果做决策，而不是靠一层文本启发式和最终 LLM 总结撑住体验。

## 1. 结论先行

SunPilot 当前主链路已经完成从多层语义决策向单一 ReAct Action/Observation Loop 的实质性迁移。前端选择的 `modelId`、`permissionMode` 能沿 `ChatComposer -> WebSocket -> JSON-RPC -> AgentService -> AgentLoopInput -> ModelRouter/PermissionPolicy` 进入首次执行；内容块流式协议、审批 checkpoint、工具安全边界、Skill 子进程隔离和 PostgreSQL 状态持久化也都已在生产装配路径中。

本轮没有发现“安全模块只实例化但未执行”这类旧问题复发，工程门禁也全部通过。但当前版本仍不能按“并发状态安全、跨端恢复完整、API/Web 行为一致”验收。本轮最重要的问题是：

1. **取消状态不是执行权闸门**：取消与首次启动、审批续跑或用户输入续跑并发时，状态管理器会静默忽略 `cancelled -> running`，但调用方仍继续模型或工具执行；
2. **`waiting_user` 只在当前页面实时事件内可恢复**：刷新页面、切换会话或 daemon 重启后，Web 无法重新绑定等待补充信息的 run；
3. **纯附件补充在 Web 可发、Core 必拒绝**：首次消息支持 attachment-only，`run.resume` 却仍要求非空文本；
4. **重试会丢失模型与权限模式**：新 attempt 默认回退到系统模型和 `auto`，与原请求不一致；
5. **工具重试没有利用 manifest 的幂等信息**：网络或写入型工具可能在“副作用已发生但响应超时”后被重复执行；
6. **REST 与 WebSocket 的响应语义不一致**：`/v1/chat` 拼接所有流式文本 delta，可能把工具前 progress 与最终回答一起返回；
7. **前端仍有鉴权、恢复和用户输入丢失问题**：artifact 内容请求未携带本地 token，发送失败后附件无法恢复，运行中会话可直接删除。

问题汇总：

| 严重度 | 数量 | 说明 |
| --- | ---: | --- |
| P0 | 1 | 可能在用户取消后继续执行模型或产生工具副作用，必须优先修复 |
| P1 | 10 | 核心行为、恢复、数据一致性或跨层契约缺陷，应进入最近修复批次 |
| P2 | 8 | 可观测性、分页、事件治理和结构性问题，建议随模块重构闭合 |

## 3. P0 问题

### P0-01：取消状态被静默忽略，已取消 Run 仍可能继续模型或工具执行

#### 证据

- `packages/core/src/agent-kernel/run-state-manager.ts:97-107`：终态收到后续状态转换时只 `console.warn` 并返回当前状态；
- `packages/core/src/agent-kernel/persistence/repository-run-state-manager.ts:72-81`：PostgreSQL 实现同样静默忽略终态转换；
- `packages/core/src/agent-kernel/agent-loop-engine.ts:104-143`：`markStatus(runId, "running")` 后不检查返回状态，直接构建上下文并进入模型；
- `packages/core/src/agent-kernel/agent-loop-engine/approval-continuation.ts:70-83`、`:116-132`、`:224-236`、`:269-276`：拒绝续跑、批准续跑、用户补充和中断恢复均在 `markStatus(..., "running")` 后直接继续；
- `packages/core/src/agent/agent-service/agent-service-implementation.ts:667-702`：`cancelRun()` 允许将 `created`、`waiting_approval`、`waiting_user` 等非终态转为 `cancelled`；
- 本轮只读运行验证确认：先将 run 标记为 `cancelled`，再调用 `markStatus(..., "running")`，返回状态仍是 `cancelled`，但不会抛错。

#### 触发方式

1. `chat.send` fast-ack 已创建 run，后台 microtask 尚未真正进入 `AgentLoopEngine.run()`；
2. 用户立即取消，run 成功变成 `cancelled`；
3. 后台协程随后执行 `markStatus(..., "running")`，该转换被静默忽略；
4. `AgentLoopEngine` 没有检查状态所有权，继续调用 LLM，甚至可能继续执行工具。

同类竞态也可能发生在审批通过/拒绝续跑和用户补充信息期间。`AbortRegistry` 能缩小窗口，但它不是持久化状态所有权；controller 被移除、重建或竞态交错时，不能替代 CAS 状态闸门。

#### 影响

- 用户看到“已取消”，后台仍可能产生模型费用、网络请求或文件/数据库副作用；
- run 记录保持 `cancelled`，实际却继续产生 tool call、artifact 或外部动作，审计语义失真；
- 后续完成状态又会被静默忽略，形成“终态正确、行为错误”的隐蔽故障。

#### 修改方案

1. 将 `markStatus()` 分成普通幂等写入和显式的 `acquireExecution(expectedStatuses)`；执行权获取必须使用数据库 CAS，并返回 `acquired: boolean` 或抛 `AGENT_RUN_STATE_CONFLICT`；
2. 首次启动只允许 `created -> running`，审批恢复只允许 `waiting_approval -> running`，补充信息只允许 `waiting_user -> running`，中断恢复只允许 `interrupted -> running`；
3. 调用方只有在成功获得执行权后才能构建上下文、调用模型或执行工具；
4. 在每个不可逆工具调用前同时检查 `AbortSignal` 和持久化 execution lease/version；
5. 新增并发回归测试：在 `markStatus` 前后插入 barrier，分别并发 cancel 与 start/approve/reject/resume，断言模型和工具执行次数均为 0。

## 4. P1 问题

### P1-01：页面刷新、切换会话或 daemon 重启后，`waiting_user` 无法在 Web 端恢复

#### 证据

- `packages/web/src/pages/ChatPage/hooks/useChat.ts:223-233`：只有收到实时 `agent.clarification.requested` 且 run 正好等于当前 `activeRunIdRef` 时，才写入 `waitingUserRunIdRef`；
- `packages/web/src/pages/ChatPage/hooks/useChat.ts:801-819`：打开持久化会话后以 `replayMissedEvents=false` 订阅，不重放 clarification 事件；
- `packages/web/src/pages/ChatPage/hooks/useConversations.ts:173-202`：历史加载只取 messages，不查询该会话的非终态 run；
- `packages/platform/src/conversations/conversation.mapper.ts:8-21`：历史消息 DTO 不携带 run 状态或 continuation runId；
- `packages/daemon/src/recovery.ts:101-117`：daemon 会保留带 checkpoint 的 `waiting_user` run，但没有重新发出可供 Web 重建状态的事件。

#### 影响

刷新后用户看到“等待补充信息”的旧消息，但下一条输入会走新的 `chat.send`，而不是 `run.resume`。旧 run 永久停留在 `waiting_user`，同一业务上下文被拆成两个 run。

#### 修改方案

- 增加 `GET /v1/conversations/:id/active-run`，返回 `created/running/waiting_approval/waiting_user/interrupted` 中的最新 run 及 checkpoint 类型；
- 选择会话和页面初始化时恢复 `activeRunId`、`waitingUserRunId`、pending approval；
- 或将 `runId`、`runStatus`、`continuationKind` 作为持久化 assistant message DTO 的正式字段；
- 补充“等待输入后刷新页面再回复”和“daemon 重启后回复”的 Web + daemon 集成测试。

### P1-02：等待补充信息时，纯附件回复在前端可发送，但协议/Core 必然拒绝

#### 证据

- `packages/web/src/pages/ChatPage/hooks/useChat.ts:841-846`：只要文本或附件任一存在即可发送；
- `packages/web/src/pages/ChatPage/hooks/useChat.ts:910-927`：`waitingUserRunIdRef` 存在时会把空文本和附件发到 `run.resume`；
- `packages/protocol/src/agent-commands.ts:75-79`：`runResumeSchema.message` 若出现必须 `.min(1)`，但没有“文本或附件至少一个”的联合校验；
- `packages/core/src/agent/agent-service/agent-service-implementation.ts:723-733`：`waiting_user` 分支明确要求 `userMessage?.trim()` 非空，不考虑附件。

#### 影响

Agent 请求用户补图、补文件或补截图时，用户无法只上传附件完成原 run；Web 先插入 optimistic 消息，随后收到 RPC 错误。

#### 修改方案

复用 `chatSendSchema` 的联合约束：`message.trim()` 非空或 `attachments.length > 0` 至少满足一个。Core 中也应使用同一 domain validator，避免 WebSocket、REST 与 Service 三套规则漂移。

### P1-03：retry/new attempt 不继承原 `modelId` 与 `permissionMode`

#### 证据

- `packages/core/src/agent-kernel/persistence/repository-run-state-manager.ts:38-42`：run 的 `input` 只保存 message、attachments、client；
- `packages/core/src/agent/agent-service/agent-service-implementation.ts:1196-1213`：`createRunAttempt()` 只恢复 mode、message、attachments，调用 `handleChatCommand()` 时没有传 modelId/permissionMode；
- 首次请求虽然在 `agent-service-implementation.ts:441-450` 正确写入 `AgentLoopInput`，但这些字段没有进入可供 retry 使用的持久化输入。

#### 影响

- 用户用 Seed + ask/full 执行失败后点击 retry，新 run 可能改用默认模型 + auto；
- 模型行为、成本和审批边界发生无提示变化；
- attempt 链不能准确重现原请求。

#### 修改方案

将完整且版本化的 `AgentRunInputSnapshot` 持久化到 `runs.input`，至少包含 `modelId`、`permissionMode`、mode、attachments、client 和原始 userMessageId；retry/resume 创建新 attempt 时显式复制，并允许 API 通过 override 字段有意识地改值。

### P1-04：工具自动重试忽略 Skill 的幂等性和 retry policy，可能重复产生副作用

#### 证据

- `packages/core/src/agent-kernel/tools/tool-types.ts:28-54`：`SkillSummary` 已有 `idempotent`、`sideEffects` 和 `timeoutPolicy.retryable/maxRetries/backoffMs`；
- `packages/core/src/agent-kernel/react-loop/tool-call-guard.ts:140-169`：构造 `PlannedToolCall` 时没有传递这些字段；
- `packages/core/src/agent-kernel/loop-types.ts:302-337`：`PlannedToolCall` 本身没有幂等/重试策略；
- `packages/core/src/agent-kernel/execution/execution-orchestrator.ts:310-329`：所有工具统一使用全局 `MAX_RETRIES` 和 `RETRY_BACKOFF`；
- `execution-types.ts:76-89`：是否重试只靠错误文本正则，如 `timeout`、`ECONNREFUSED`、`rate limit`。

#### 影响

网络发布、创建订单、写文件等非幂等动作可能已经在外部成功，只是客户端在取响应时超时。当前实现会再次执行同一动作，产生重复发布、重复写入或重复业务操作。

#### 修改方案

- 将 `idempotent`、`sideEffects`、`timeoutPolicy` 固化到 checkpointed `PlannedToolCall`；
- 默认只自动重试 `idempotent=true` 或 readonly 工具；mutation/network/destructive 必须由 Skill 提供 idempotency key/查询恢复能力；
- 使用结构化错误码决定 retry，停止使用 message regex 作为主要依据；
- 每次 attempt 单独记录 attempt number、startedAt、error 和外部 request id。

### P1-05：REST `/v1/chat` 返回的是全部流式 delta 拼接，不是 Core 的最终回答

#### 证据

- `packages/api/src/http/register-routes.ts:142-180`：`assistantContent += delta.delta`，最后把该字符串作为 assistant message 返回；
- `packages/core/src/agent-kernel/react-loop/react-model-turn.ts:101-115`：工具前 progress 文本同样会产生 `agent.message.part.delta`；
- `packages/core/src/agent-kernel/assistant-message-stream.ts:659-679`：Core 持久化时只把 `semanticRole=final/user_prompt` 合并为正式 content。

#### 影响

同一请求通过 WebSocket 与 REST 得到不同答案。发生工具调用时，REST 可能把“我先查询……”等 progress、多个模型轮次文本和最终回答连在一起，甚至暴露本应只在折叠思考区展示的内容。

#### 修改方案

让 `AgentLoopResult` 明确返回最终 `content`，或在 loop 完成后读取 `assistantMessageId` 对应的已持久化 message；REST 不应自行从 transient delta 重建业务结果。若需要 SSE，应该输出结构化 content-block 事件而不是最后再拼字符串。

### P1-06：Artifact 内容请求绕过统一请求客户端，启用 token auth 后打开产物会 401

#### 证据

- `packages/web/src/features/agent-runtime/api.ts:72-79`：metadata 使用注入的 `request`，content 却直接 `fetch('/v1/artifacts/.../content')`；
- `packages/web/src/shared/api/client.ts:3-16`：只有统一 `createRequest()` 会注入 `Authorization: Bearer <localToken>`；
- `packages/daemon/src/server.ts:424-445`：`/v1/*` 在 token auth 开启时必须携带有效 bearer token。

#### 影响

Artifact 列表和元数据可加载，但用户点击内容时失败；现有测试在禁用 token auth 的环境下无法发现。

#### 修改方案

让 `getAgentArtifactContent(request, artifactId)` 也使用统一 client，并按 blob/text MIME 类型处理；新增 token auth 开启状态下的前端集成测试。

### P1-07：运行中的会话可以直接删除，后台 run 随后因消息/事件目标消失而失败

#### 证据

- `packages/web/src/layouts/AppShell/RecentConversations.tsx:145-180`：所有会话始终显示删除入口，不关心 active run；
- `packages/platform/src/conversations/conversation.service.ts:122-130`：删除前不检查非终态 run，也不触发 cancel；
- `packages/storage/src/postgres/postgres.conversation.repository.ts:98-119`：直接删除 messages、events、trace 和 conversation；run 的 FK 是 `ON DELETE SET NULL`；
- Agent Loop 后续仍会使用原 `conversationId` 持久化 assistant message 和事件。

#### 影响

外部工具可能已经执行成功，但最终 assistant message 保存因 conversation 不存在而失败；run 与 conversation 脱钩，事件历史也被提前删除。

#### 修改方案

删除会话前事务内查询非终态 run：默认返回 409 并提示先停止，或先 cancel 并等待 execution lease 释放后再删。Web 在 active run 存在时应禁用删除或展示明确的“停止并删除”。

### P1-08：发送失败后附件无法恢复，attachment-only 请求会完全丢失用户输入

#### 证据

- `packages/web/src/pages/ChatPage/components/ChatComposer/ChatComposer.tsx:135-136`：所谓“保存附件”实际只存了 `hasFiles: boolean`；
- `ChatComposer.tsx:161-182`：失败恢复只恢复非空文本；
- `ChatComposer.tsx:239-266`：发送后立即 `clearFiles()`；
- attachment-only 消息的 `lastSentRef.current.text` 为空，失败 effect 不会执行任何恢复。

#### 影响

网络失败、RPC 校验失败或 daemon 断线后，用户需要重新选择/上传附件；纯图片请求会同时丢失附件和输入上下文。

#### 修改方案

在发送前保存完整的不可变 draft snapshot（text、AttachmentRef、UploadFile 显示信息、model、permission），收到 ack 后再清除，或失败时从 snapshot 原样恢复。queued upload 也应快照“点击发送那一刻”的文本和选项。

### P1-09：HTTP 与 WebSocket 的审批拒绝默认策略不一致

#### 证据

- `packages/core/src/agent/agent-service/agent-service-implementation.ts:943-948`：Service 默认 `continue_without_tool`；
- `packages/protocol/src/agent-commands.ts:100-107`：JSON-RPC `approval.reject` 默认 `interrupt`；
- `packages/api/src/http/register-routes.ts:690-700`：HTTP route 不解析/传递 strategy，因此走 Service 默认继续；
- `packages/web/src/features/agent-runtime/api.ts:55-59`：Web 拒绝按钮使用 HTTP route，不携带 strategy。

#### 影响

同一个“拒绝”动作经 Web UI/HTTP 会继续让模型寻找替代方案，经 JSON-RPC 却会中断 run。客户端无法依赖统一语义，前端提示也无法准确说明结果。

#### 修改方案

只保留一套 `ApprovalRejectionPolicy`：协议 schema、HTTP body、Service 必须共享默认值；Web UI 应明确提供“拒绝并继续”“拒绝并停止”或固定一种产品语义。

### P1-10：删除会话后的前端 next conversation 选择依赖 React updater 的执行时机

#### 证据

- `packages/web/src/pages/ChatPage/hooks/useConversations.ts:125-142`：`nextId` 在 `setConversations()` 的 functional updater 内赋值，却在调用 `setConversations()` 后立即读取。

React 不保证 updater 在当前语句之后同步执行。在并发渲染或批处理下，`nextId` 仍可能是空字符串，于是即使还有会话也会错误清空 active conversation 和 messages。

#### 修改方案

不要从 state updater 向外写临时变量。可维护 `conversationsRef`，删除成功后先计算 `remaining/nextId`，再分别 set；或者用 reducer 在一个 action 中原子更新 conversations、activeConversationId 和 messages。

## 5. P2 问题

### P2-01：ReAct 拒绝续跑仍冻结原候选工具集，替代方案可能永远不可见

- `react-loop-runner.ts:117-158` 在 rejection continuation 中继续传 `candidateToolIds`；
- `react-loop-runner.ts:247-264` 发现 frozen IDs 后不再重新检索；
- 对比 `resumeWithUserInput():196-203`，用户补充路径有意识地清除了候选集。

拒绝通常意味着需要改变执行方案。建议 rejection 后重新检索候选集，同时保留已拒绝 action signature，防止模型重新申请同一动作。

### P2-02：Web 的 canonical event visibility 表缺少 `agent.react.turn.completed`，却保留已退出主路径的旧事件

- Protocol 已在 `packages/protocol/src/agent-events.ts:53` 定义 `agent.react.turn.completed`；
- `packages/web/src/pages/ChatPage/hooks/chat/chat-state.ts:571-612` 的表没有它；
- `activityFromAgentEvent()` 对未映射事件会打印 warning；
- 表中仍保留 `agent.intent.detected`、`agent.plan.created`，与当前 ReAct 主路径不一致。

建议直接从 Protocol 导出强类型 visibility map 或建立编译期 exhaustive assertion，新增事件时让 Web typecheck 失败，而不是运行时 warning。

### P2-03：前端复制了一整套 AssistantMessagePart 类型，已经出现 metadata 枚举漂移

- `packages/protocol/src/message-parts.ts:17-100` 是 canonical 定义；
- `packages/web/src/features/conversations/types.ts:22-104` 又复制一份；
- Protocol status phase 包含 `summarizing`，Web 副本包含 `local_pending`，两边并不相同；
- Web 已依赖 `@sunpilot/protocol`，没有继续复制的必要。

建议 Web 直接 `import type` canonical parts；本地 optimistic 字段用单独的 `LocalPendingPart` union 扩展，不污染服务端协议类型。

### P2-04：Conversation 游标未包含首排序键 `pinned`，分页可能跳项

- `postgres.conversation.repository.ts:42-55` 按 `pinned DESC, updated_at DESC, id DESC` 排序；
- `:124-142` 的 cursor 只有 updatedAt/id；
- 下一页 WHERE 也只比较 updatedAt/id。

当一页跨越 pinned/unpinned 边界时，部分更新时间较新的普通会话可能被跳过。cursor 应包含 pinned，并使用完整 tuple 条件；Platform 和 API 的 cursor encoder 也应共享同一 codec。

### P2-05：StaleDetectionWorker 每轮只处理 50 条且 stop 不会取消首次延迟扫描

- `packages/daemon/src/stale-detection-worker.ts:30-32` 的 initial timeout 未保存；`stop()` 只清 interval；
- `:50-54` 每轮固定搜索 50 条，没有游标或稳定扫描水位。

结果是 daemon 已停止后仍可能触发一次 DB scan；数据量增长后，排序靠前的 50 条还可能长期占住窗口。建议保存并清理 initial timer，使用 cursor/batch watermark 扫描全部候选。

### P2-06：事件持久化失败被吞掉，Run 可成功但审计/重放事件缺失

- `packages/daemon/src/factories/persistence-factory.ts:60-91` 将持久化错误记录日志后吞掉；
- lifecycle start 还会先无 sequence 推送到 live bus，再 best-effort 持久化；
- 业务主流程无法知道 `agent.run.completed`、approval、tool lifecycle 等事件是否真正落库。

建议把事件分为 durable-required 与 transient。终态、审批和安全事件必须进入与状态写入同一事务或 outbox；delta 才允许 best-effort。监控中应暴露 queue depth、persist failure 和 oldest pending age。

### P2-07：可观测性字段把字符数当 token 数

- `react-model-turn.ts:192-200` 的 `agent.model.completed.outputTokens` 直接使用 `text.length`；
- `ModelRouter` 又用 `Math.ceil(chars/4)` 记录估算 token。

同一模型调用在 event 与 model_calls 中口径不同。建议字段改名 `outputChars`，或统一使用 provider usage/同一 tokenizer estimate，并标记 `estimated: true`。

### P2-08：Pending approval 是全局加载，当前会话会显示其他会话的审批

- `packages/web/src/features/agent-runtime/api.ts:44-45` 请求所有 pending approval；
- `ChatPage.tsx:172-176` 不按 active conversation/run 过滤直接渲染；
- approval DTO 只有 runId，前端没有同步加载 run->conversation 映射。

建议 `/v1/approvals` 支持 `conversationId`，Chat 页面只展示当前会话；全局审批中心可作为独立入口，避免误操作其他任务。

## 6. Agent Core 专项评价

### 6.1 做得较好的部分

1. `ReactLoopRunner` 已成为下一步语义决策的唯一权威，旧的 IntentRouter/Planner/Replanner/ResponseComposer 不再控制主路径；
2. Tool retrieval 只提供候选能力，Guard 负责 schema/permission/重复调用，ExecutionOrchestrator 负责唯一执行边界，职责方向正确；
3. tool output 会转换为 observation，协议错误也会回注模型而不是直接让进程崩溃；
4. approval/user-input/interrupted 已有持久化 checkpoint 和 transcript 连续性；
5. 工具执行前后都经过 `ToolSafetyBoundary`，审批 scope 与参数快照也进入执行校验；
6. `AssistantMessageStream` 将 progress、final、user_prompt 分离，Core 持久化 content 与 Web 产品展示的方向已经统一。

### 6.2 仍需优先优化的函数结构

#### `AgentServiceImplementation` 过度聚合

文件：`packages/core/src/agent/agent-service/agent-service-implementation.ts`，约 1400 行。

当前同时负责 fast-ack、幂等、conversation/message/run 创建、事件订阅、abort、cancel、resume、retry、approve/reject、attempt link 和错误映射。建议拆为：

- `ChatCommandStarter`：校验、幂等预留、conversation/run/message 原子初始化；
- `RunExecutionSupervisor`：background execution、abort、event flush、terminal cleanup；
- `RunContinuationService`：waiting_user/interrupted checkpoint 恢复与 execution lease；
- `ApprovalDecisionCoordinator`：approve/reject/expiry 与统一 rejection policy；
- `RunAttemptService`：retry/resume-new-attempt 和完整 input snapshot 继承。

#### `ApprovalContinuationCoordinator` 与 `AgentLoopEngine` 重复终态逻辑

两处都实现 waiting_approval、waiting_user、stream.complete、memory write、run completed/failed/cancelled event。当前重复已导致状态处理和错误 category 容易分叉。

建议提取 `ReactRunSession.finalize(result)`，统一：

- stream snapshot/complete；
- checkpoint save/clear；
- memory write；
- run CAS terminal transition；
- terminal event + flush；
- safety grant cleanup。

#### `ContextAssemblyPipeline` 过长且边界模糊

`context-assembly-pipeline.ts` 超过 1300 行，混合并行 IO、summary、memory retrieval、token budget、压缩、日志和 event。建议拆为 source loaders、trust normalizer、budget planner、context packer；用结构化 logger/metrics 替代散落的 `console.*` 和 `as any`。

#### 重试策略应属于 capability policy，而不是 ExecutionOrchestrator 全局常量

`SkillSummary.timeoutPolicy` 已有设计基础，应让 Guard 冻结为执行 policy，Orchestrator 只消费。这样测试可以针对 readonly、idempotent mutation、non-idempotent mutation 分别验证。

## 7. 前端与 Agent Core 联动专项评价

  -X-> waiting_user run 重建

## 10. 建议验收清单

- cancel 与首次启动并发：0 次模型调用、0 次工具执行；
- cancel 与 approval approve/reject 并发：0 次未授权续跑；
- cancel 与 waiting_user resume 并发：只允许一个状态所有者；
- 用户只上传图片回答 clarification：同一 run 正常继续；
- waiting_user 页面刷新、切换会话、daemon 重启：均能恢复原 run；
- retry 保留 modelId、permissionMode、attachments 和 client metadata；
- 非幂等 mutation 工具遇到 timeout：不自动二次执行；
- `/v1/chat` 返回内容与数据库 assistant message content 完全一致；
- token auth 开启时 artifact metadata/content 都能加载；
- 运行中删除会话返回 409 或完成 cancel-and-delete；
- HTTP/WS 拒绝审批得到相同策略和事件序列；
- conversation pinned/unpinned 跨页无重复、无遗漏；
- 所有 canonical agent event 在 Web 有编译期可穷尽映射；
- `pnpm typecheck && pnpm lint && pnpm test && pnpm build` 全部通过；
- `pnpm audit --prod --registry=https://registry.npmjs.org` 无已知漏洞。

## 11. 最终判断

SunPilot 的 Agent Core 重构方向是正确的：LLM 已成为 ReAct 下一动作的唯一语义权威，代码侧负责候选、校验、安全、执行、状态和 observation。当前最需要避免的是把“主循环结构正确”误判为“运行语义已经稳定”。

本轮发现的核心风险集中在**执行权所有权、恢复身份、跨端契约和副作用重试**。其中 P0-01 应先于任何 UI 优化处理；如果取消不能可靠阻止后续动作，那么状态机、审计事件和前端展示即使都正确，也只是在准确展示一个不安全的执行系统。
